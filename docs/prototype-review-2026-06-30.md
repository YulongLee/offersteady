# Web 原型功能审查（2026-06-30）

状态：Draft

本次审查基于 `recover-local-prototype-and-review-product-experience` 变更执行，目标是先确认本地原型是否仍可作为产品验证基线，再区分需要立即恢复的问题与后续单独决策的问题。

## 审查范围

- 首页
- 登录/注册
- 面试首页
- 资料页
- 面试准备页
- 实时面试页
- 复盘页
- 积分页
- 设备下载页
- 使用说明页
- 设置页

## 立即恢复项

1. 首页文案仍残留旧的“说话人不确定时先让你确认”表达，与当前双角色实时页方案不一致。
2. 使用说明中的快速开始仍写旧的开始门槛，与现行“确认本场资料即可进入面试；本地端负责输入检查”的准备流程不一致。
3. 本地自动巡检需要明确依赖预览服务；若直接运行巡检脚本，会把服务未启动误判为页面异常。

## 本次已恢复

- 首页实时辅助文案改为：来源不清时暂停自动回答并提示改用手动输入。
- 快速开始文案改为：确认本场资料后即可进入面试，音频与问题检测由本地端检查。
- 补充受保护原型基线文档，避免后续工程接入继续顺手改动页面体验。

## 暂未发现需要立即改动的路径

- 登录主入口仍为微信登录/注册，开发态保留本地原型用户入口
- 面试首页仅保留“继续面试”
- 准备页未恢复通用数据用途复选框
- 实时面试页保持双角色、左右分栏、底部紧凑操作与终止回答
- 资料页仍区分简历、JD、知识库，并展示上传格式提示
- 积分页保留兑换码、积分说明与官方支付流程
- 设置页未恢复“面试记录保存期限”下拉项

## 后续单独决策项

1. Windows 下载入口当前仍按签名状态展示“签名验证中/完成签名后开放”；若要对外宣称“已支持 Windows”，需要先在发布清单、签名状态和使用说明上同步成真实可下载状态，再单开变更处理。
2. `docs/README.md` 仍保留旧的“具体功能需求应放入 specs/”说法，和当前 OpenSpec 主路径不完全一致；这属于文档治理问题，不影响原型恢复，可后续单独收敛。

## 建议的本地验证顺序

1. `npm run build -w @offersteady/web`
2. 启动 Web 预览服务（如 `npm run dev:web` 或 `vite preview`）
3. 运行 Web 自动巡检与手动路径检查
4. 确认首页、准备页、实时页和积分页在手机/桌面下都可访问
