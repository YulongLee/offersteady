# Web 原型受保护基线

状态：Active

本文档记录当前已恢复并需要保护的 Web 原型基线。后续接入后端、协议或工程能力时，默认不得静默改变这些页面结构、主路径和交互顺序；如需调整，应先新增并批准对应 OpenSpec 变更。

## 受保护的主路径

```text
首页 -> 微信登录/注册 -> 面试首页 -> 资料页/新建面试 -> 准备页 -> 实时面试 -> 复盘页
                          \-> 积分页 -> 设备下载页 -> 使用说明页 -> 设置页
```

## 必须保持不变的体验约束

### 1. 首页

- 首页以用户收益和使用节奏为主，不回退到 `CLEAR BOUNDARIES` 一类内部边界型主卖点。
- 首屏核心价值必须同时覆盖：
  - 实时理解问题并生成回答思路
  - 结合简历、JD 和知识材料进行个性化辅助
  - 支持按次积分或按天会员两种节奏
- 首页不得引入旧的“角色待确认”产品心智；对来源不清的情况只说明会暂停自动回答并提示改用手动输入。

### 2. 登录与首页恢复入口

- 登录页主入口为微信登录/注册；开发态可保留本地原型入口。
- 面试首页针对未结束会话只保留一个“继续面试”主入口，不恢复“继续准备”或“预览工作台”并行入口。

### 3. 资料页与准备页

- 资料页必须区分简历、JD、知识库三个分区。
- 上传说明需明确支持的格式：PDF、DOCX、DOC、TXT、MD。
- 准备页使用与资料页同一来源集合；本场资料允许为空。
- 准备页不恢复通用“我已了解数据用途”复选框。
- 准备页确认本场资料后即可进入面试；本地端收音、系统音频和问题检测检查不得阻塞 Web 端进入实时页。

### 4. 实时面试页

- 只展示两个角色：“面试官”和“我”。
- 桌面端保持左右两栏：实时对话在左、回答在右，并支持拖动分栏。
- 手机/窄屏保持“对话 -> 回答 -> 紧凑操作栏”的垂直顺序。
- 页面不恢复资料侧栏、角色修正按钮、历史侧栏或单独的现场资料面板。
- 手动输入、“回答问题”和“截图回答”保持在回答区底部。
- 回答生成中必须提供“终止回答”。

### 5. 积分、支付与使用说明

- 积分页保留兑换码入口、积分消费说明、会员档位和官方支付说明。
- 支付流程不恢复手工上传付款截图、填写交易单号或人工检验文案。
- 使用说明中的流程描述必须与当前准备页和实时页保持一致。

## 当前恢复结论

本轮检查中，原型代码本身未出现编译失败、测试失败或构建失败；“本地原型崩掉”主要表现为：

- 本地巡检依赖预览服务，未先启动服务时会被误判为首页异常
- 首页与使用说明存在少量旧文案，已与现行流程不一致

这些问题都属于应恢复到基线的原型偏移，而不是新增功能需求。
