## Context

OfferSteady 当前仍处于产品原型验证阶段，`apps/web` 里的页面结构、文案表达和交互节奏本身就是被用户逐步确认下来的产品资产。最近在推进统一 Document Service 和后端联调时，资料页与上传适配层被联动修改，虽然工程目标是接入能力，但结果触碰了原型层稳定性。

这次变更的目标不是新增功能，而是恢复并保护“已批准的 Web 原型表现”。它需要明确哪些部分属于原型层真相，哪些部分属于后端能力适配层，并要求后续后端接入优先通过兼容层完成，而不是直接改变页面表现。

受影响最明显的区域是：

- `LibraryManager.tsx` 中的资料管理展示、提示文案和删除行为
- `material-upload-adapter.ts` 中原型模式与 API 模式的兼容方式
- 资料上传、知识库资料列表和状态展示相关的原型节奏

## Goals / Non-Goals

**Goals:**

- 恢复已批准的 Web 原型表现，不因后端联调改变页面交互和文案
- 明确原型层与后端能力适配层的边界
- 为后续能力接入建立兼容约束：接新接口时尽量不改变页面表现
- 对资料页和上传流程建立“原型完整性优先”的规则

**Non-Goals:**

- 不回退已经确认需要保留的后端 Document Service 基础能力
- 不重新设计资料页、面试页或计费页产品方案
- 不否定后端统一文档服务本身的方向，只约束其接入方式
- 不把这次变更扩展为完整的版本控制或仓库治理方案

## Decisions

### 1. Treat approved web prototype behavior as a protected product baseline

今后已批准的 Web 原型页面表现应被视为受保护基线。工程实现可以调整内部适配层，但不得默认改变：

- 页面结构
- 文案语气
- 交互顺序
- 原型模式下的演示反馈

原因：

- 当前阶段产品验证比后端联调更重要
- 用户已投入时间反复调过原型，不应被基础工程推进反复冲掉

替代方案是继续把原型当作临时壳，允许后端接入时顺手修改。这样会让产品验证持续失真。

### 2. Keep backend capability evolution, but isolate it behind compatibility adapters

后端可以继续演进统一 Document Service、PostgreSQL 元数据和 OSS 边界，但 `apps/web` 应优先通过兼容 adapter 消化这些变化，而不是直接让页面逻辑跟随后端接口形状改变。

原因：

- 原型模式和 API 模式可以共享外观，但不需要完全共享内部实现节奏
- 后端升级不应立刻改变前端原型的删除、刷新和状态体验

替代方案是让页面组件直接依赖最新后端 contract。这样最省工程层，但会持续损伤原型稳定性。

### 3. Scope the rollback to prototype-facing files first

本次回退应优先聚焦于用户可感知的原型层文件，例如：

- `apps/web/src/LibraryManager.tsx`
- `apps/web/src/material-upload-adapter.ts`
- 如有必要，再补充少量直接影响资料页表现的文件

原因：

- 用户最关心的是“我辛苦搭建的原型是否被改掉”
- 快速回退原型层能先恢复信任和可演示状态

替代方案是连同后端一起整体回退。这样风险更大，也会把已完成的服务端基础能力一起抹掉。

## Risks / Trade-offs

- [Risk] 原型层恢复后，前后端联调行为可能再次出现分叉 → Mitigation: 明确 adapter 是兼容层，后续接入要先保持原型表现一致
- [Risk] 只回退前端不回退后端，会留下部分“能力已实现但页面未使用”的代码 → Mitigation: 接受短期冗余，以保护产品原型优先
- [Risk] 后续多人开发时再次误动原型层 → Mitigation: 通过本次 spec 把“原型完整性优先”写成明确要求

## Migration Plan

1. 定义“已批准 Web 原型完整性”能力规范。
2. 回退资料页和上传适配层中用户可感知的误改。
3. 验证原型模式和基础页面表现恢复正常。
4. 保留后端 Document Service，但要求后续联调通过兼容适配层逐步接入。

回滚策略：

- 如果回退过程中发现某些后端兼容字段是前端编译所必需，则仅保留最小兼容字段，不保留新的页面行为
- 若后续需要再次调整原型表现，必须通过新的 OpenSpec 变更，而不是在实现型 change 中顺手修改
