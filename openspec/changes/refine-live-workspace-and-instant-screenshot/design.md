## Context

当前实时面试页已经采用“左侧实时对话 + 右侧回答”的双栏主结构，但会话控制仍停留在底部栏，手动问题输入和截图回答操作也没有按用户视线与操作路径分布到对应主区。结果是用户在面试中需要频繁跨区找按钮，底部栏也会和阅读、输入、软键盘或安全区产生冲突。

同时，现有截图回答原型仍然沿用上传式心智，包含预览或上传中间态，这与真实面试中的“看到题就立刻截屏求答”不一致。当前 change 需要在不扩展额外页面的前提下，把实时面试页改成更接近实战节奏的布局，并把截图回答抽象成即时屏幕捕获入口。

## Goals / Non-Goals

**Goals:**
- 保持实时页双栏主结构，但重新分配控制区、输入区和回答触发区
- 将“开始面试 / 结束面试”移动到右上角头部区域，减少底部主区干扰
- 将手动问题输入框稳定放在左侧实时对话区下方
- 将“快答 / 截屏回答”稳定放在右侧回答区下方
- 将截图回答改成用户主动触发的即时屏幕捕获链路，而不是文件上传链路
- 让桌面与窄屏布局都能维持一致的认知模型和可访问性

**Non-Goals:**
- 不改变准备页、本场资料选择或积分页主流程
- 不新增独立的截图管理页面或上传中心
- 不在本 change 中扩展新的截图标注、裁剪或多图拼接能力
- 不修改 Screenshot Answer Service 的模型选择策略，只调整实时页如何发起该能力

## Decisions

### 1. 将会话控制上移到头部右侧，而不是继续留在底部栏
我们将“开始面试 / 结束面试”放入实时页头部右上角，与当前会话标题、状态和积分入口形成同一操作层。这样用户在阅读对话和回答时，主内容底部只保留和内容直接相关的输入或触发动作。

选择原因：
- 会话级控制属于全局状态，不应与问题级回答动作混在同一底部区域
- 头部区域在桌面和手机都更容易形成稳定预期

备选方案：
- 保留底部栏，仅缩小按钮尺寸
  未选择，因为即便缩小，仍会继续打断用户在主内容区的阅读与输入路径

### 2. 将实时页底部操作拆成左右各自所属的局部操作区
左栏下方只保留手动问题输入框，右栏下方只保留“快答 / 截屏回答”触发动作。窄屏时按“对话 -> 输入 -> 回答 -> 动作”单列展开，保持桌面逻辑的顺序投影。

选择原因：
- 手动输入与实时对话是同一心智链路，靠近左栏更顺手
- 快答和截屏回答本质上是在驱动右侧回答区生成新内容，应靠近回答区
- 这比一个通用底部混合操作栏更清晰

备选方案：
- 继续使用统一紧凑底栏
  未选择，因为会把“输入问题”和“触发回答”耦合在一个区块里，用户定位成本更高

### 3. 将截图回答改为 Capture Adapter，而不是上传式文件输入
实时页的“截屏回答”入口将面向一个可替换的 Capture Adapter。Web 端优先调用浏览器/平台的当前屏幕捕获能力，桌面端后续可接入本地伴随程序提供更顺滑的截图桥接；上层 UI 不再暴露上传式文案或文件预览流程。

选择原因：
- 能保持 Screenshot Answer Service 作为下游统一服务边界
- Web 与桌面端可以共用同一上层产品交互，只替换采集实现
- 后续接入更稳定的本地截屏能力时，不需要再次重做页面结构

备选方案：
- 直接在页面里继续使用文件上传控件
  未选择，因为这与用户明确要求的现场即时截屏场景冲突

### 4. 失败恢复继续停留在实时页，不引入额外模态流程
截屏权限失败、用户取消、识别失败或回答失败时，系统继续留在当前实时面试页，只展示轻量失败状态和重新触发入口；不进入上传补救页，也不自动重试截屏。

选择原因：
- 面试中用户没有时间在额外流程里来回切换
- 失败恢复需要可预期且最少步骤

备选方案：
- 失败后自动弹出上传本地图片作为兜底
  未选择，因为会把即时截屏链路再次拉回上传式心智

## Risks / Trade-offs

- [Risk] 右上角同时放会话状态、积分和开始/结束控制，头部密度会变高 → Mitigation: 采用紧凑分组和清晰主次层级，避免把问题级按钮也塞进头部
- [Risk] 窄屏顺序从“对话 -> 回答 -> 操作栏”改成“对话 -> 输入 -> 回答 -> 动作”，会改变既有滚动节奏 → Mitigation: 在 spec 中明确新顺序，并在 apply 时补移动端回归测试
- [Risk] 浏览器或平台的屏幕捕获能力存在权限差异 → Mitigation: 通过 Capture Adapter 屏蔽实现差异，并为拒绝权限、取消捕获和不可用状态定义统一恢复文案
- [Risk] 即时截屏减少了预览确认，会提升误截风险 → Mitigation: 截图必须继续由用户主动触发，取消或权限失败时不写入会话，失败重试需要重新触发

## Migration Plan

1. 先调整实时页布局契约和组件职责，不改后端回答能力边界
2. 再把现有截图回答入口从上传式状态机切换为即时捕获状态机
3. 为桌面和窄屏补充布局与交互测试，覆盖开始/结束控制位置、左右区底部操作、即时截屏取消与失败恢复
4. 若即时截屏在某平台不可用，可回滚到旧入口实现，但不应保留旧上传文案作为默认体验

## Open Questions

- Web 端首发是否只支持“整屏截取”，还是需要在浏览器可用时允许选择窗口或标签页
- 若桌面伴随程序已连接，是否应优先使用本地端截屏能力而不是浏览器捕获能力
