## Why

当前实时面试回答虽然已经接入真实模型、简历/JD 固定上下文和知识库 RAG，但简单回答与详细回答由两段硬编码策略分别生成，题型、语气、事实依据和展开重点缺少统一规划，容易出现内容重复、表达生硬、资料使用不明显或回答没有直接命中问题。截图回答也只有通用格式约束，缺少针对代码、SQL、系统设计、报错和信息不完整截图的稳定答题策略。

现在需要建立一套可版本化、可评测、可回滚的提示词工程体系，在不改变产品原型和模型供应商的前提下，提高回答的直接性、口语感、岗位匹配度、事实可信度和不同题型下的稳定性。

## What Changes

- 为实时面试回答增加问题意图与题型自适应策略，覆盖自我介绍、项目经历、行为面试、技术原理、系统设计、岗位匹配、压力追问和信息不足等常见场景。
- 让简单回答与详细回答共享同一份事实计划和回答主张，简单回答负责快速开口，详细回答负责自然展开，避免两段互相矛盾或机械重复。
- 明确简历、JD、知识库和多轮对话的优先级与使用边界：简历/JD提供候选人事实，JD提供岗位关注点，RAG只补充命中的知识依据，任何来源都不得被模型扩写成不存在的经历或数字。
- 为截图回答增加题型路由、答案完整性、编程语言选择、代码可运行性、复杂度、截图不完整和无法可靠识别时的保守策略，同时继续禁止使用个人资料或知识库。
- 将运行时硬编码的回答策略迁移到 `ai/prompts/` 下的版本化模板或受控片段，统一记录 prompt template、版本和策略模式，支持快速回滚。
- 扩充 `ai/evals/`，使用合成场景覆盖资料一致性、直接回答、自然口语、追问衔接、题型适配、代码完整性、无资料不编造以及简单/详细回答一致性。
- 增加提示词版本离线评测与最小线上质量遥测；不记录原始简历、JD、知识库正文、截图、完整问题或完整模型回答。

## Capabilities

### New Capabilities

- `adaptive-interview-answering`: 定义实时面试回答如何识别问题意图、选择回答结构，并保持简单回答与详细回答一致、自然和可直接口播。
- `grounded-answer-evidence-policy`: 定义简历、JD、知识库和多轮对话在回答中的事实优先级、引用边界、冲突处理和无资料退化行为。
- `prompt-quality-governance`: 定义提示词集中管理、版本记录、评测门槛、隐私安全遥测和回滚要求。
- `adaptive-screenshot-answering`: 定义截图回答针对不同题型的输出质量、完整性和无法可靠识别时的保守行为。

## Impact

- Prompt 与评测：`ai/prompts/chat-service/`、`ai/prompts/screenshot-answer/`、`ai/evals/`
- 后端：Chat Service 的 prompt 组装、简单/详细回答生成协调、资料上下文注入、截图视觉提示词、prompt 版本与质量遥测
- 测试：回答结构、题型路由、资料 grounding、无编造、截图代码完整性和 prompt 回滚回归测试
- API：保持现有回答接口与页面结构兼容；仅在需要时增加非敏感的 prompt strategy/version 元数据
- 隐私：不得在日志、评测夹具或遥测中保存真实简历、JD、知识库正文、截图、完整转录或模型完整回答
- 非目标：不更换当前模型供应商，不改变 RAG Top-K 和计费规则，不改变页面原型，不让截图回答使用个人资料，不新增自动代答或隐蔽使用能力
