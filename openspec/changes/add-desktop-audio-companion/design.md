## Context

OfferSteady 的业务界面应统一由响应式 Web 应用提供，让用户在电脑和手机浏览器中访问同一场面试。浏览器适合展示资料和回答，但系统音频采集、持续权限状态以及跨平台音频设备管理需要一个可下载安装的桌面伴随程序。

桌面伴随程序不是第二套产品界面。它只负责授权绑定、音频源选择、采集状态、音量诊断和安全传输；问题、转录、回答和设备状态仍由服务端同步到 Web 页面。

## Goals / Non-Goals

**Goals:**

- 电脑和手机继续通过同一个 Web 应用完成主要产品流程。
- 提供轻量、可见、可随时停止的桌面音频采集程序。
- 支持麦克风和受操作系统允许的系统音频输入，并保留平台适配边界。
- 将桌面采集程序安全绑定到当前面试会话。
- 让任意已授权 Web 设备看到采集、转录和连接状态。
- 默认不长期保存原始音频，并支持立即撤销桌面设备。

**Non-Goals:**

- 不在桌面程序中复制简历、JD、知识库和完整回答页面。
- 不提供隐蔽模式、静默启动、绕过权限或无法被用户察觉的后台监听。
- 不保证所有操作系统版本都能采集系统音频。
- 不在 MVP 中实现离线 AI 回答或本地大模型。
- 不默认保存完整面试录音。

## Decisions

### 1. Web 是唯一业务前端，桌面程序是薄伴随层

```text
电脑 Web ─┐
          ├── 会话 API / 实时状态 ── 问题识别与回答管线
手机 Web ─┘              ↑
                         │ 音频/转录事件
桌面伴随程序 ─────────────┘
  ├── 麦克风适配器
  ├── 系统音频适配器
  └── 权限、状态与安全传输
```

Web 页面提供“下载桌面伴随程序”“连接本机”“查看采集状态”和“撤销设备”。桌面程序只保留最小窗口或菜单栏/托盘入口。

备选方案是构建完整桌面产品。这样会产生 Web 与桌面两套导航和状态逻辑，增加迭代成本，因此不用于 MVP。

### 2. 按平台实现可替换的音频采集适配器

统一采集接口输出带来源、时间戳、序列号和格式信息的音频帧。首发实现 macOS 14.2+ 的麦克风和系统音频适配器；不支持的来源必须显示明确原因，并允许退化为麦克风或手动问题输入。

首发安装包使用 universal 架构，同时包含 Intel x64 与 Apple Silicon arm64。平台能力不通过伪造统一行为来掩盖，而是由能力检测结果驱动界面。Windows 作为第二阶段适配器实现。

### 3. 麦克风与系统音频保持独立声道

在平台允许时，候选人麦克风与会议系统音频分别采集、编码和传输。独立声道有利于区分说话方、避免回声和提高问题识别质量；服务端可以在无法分离时退化为混合输入。

### 4. 通过 Web 发起短期设备绑定

用户在 Web 会话中点击“连接桌面收音”，服务端签发短期、一次性绑定凭证。Web 使用自定义深链唤起已安装程序，或展示配对码供用户粘贴。桌面程序交换得到仅限当前会话和音频权限的短期设备凭证。

设备凭证与用户主登录凭证分离，撤销桌面设备不会退出用户的 Web 会话。

### 5. MVP 使用安全持久连接传输有序音频帧

桌面程序通过出站加密连接发送压缩音频帧和状态事件。帧包含会话 ID、设备 ID、来源、序列号和时间戳；服务端按来源保持顺序并丢弃重复帧。传输层通过接口封装，后续可以在延迟或网络质量需要时替换为 WebRTC 等方案。

为控制复杂度，MVP 优先采用单一持久连接协议，而不是同时实现多种传输。

### 6. 原始音频以流式短暂处理为默认

桌面端只保留有界的内存缓冲用于网络抖动恢复，不写入本地录音文件。服务端将音频交给可替换的实时转录适配器，默认不持久化原始音频；转录文本和问题是否保留遵循会话数据策略。

### 7. 采集状态必须在桌面与 Web 同时可见

桌面程序始终展示 `未连接`、`需要权限`、`已就绪`、`采集中`、`已暂停`、`重连中` 或 `错误`。采集中时使用持续可见的状态指示；Web 端同步展示桌面设备和输入来源。关闭、暂停或撤销任一入口都必须停止后续音频发送。

### 8. 下载与版本兼容由 Web 协调

Web 根据浏览器报告的平台展示受支持安装包，并同时提供手动选择。桌面程序绑定时报告版本和能力；服务端可以拒绝不兼容版本，并返回升级说明。正式发布需要签名安装包和可验证的下载来源。

## Risks / Trade-offs

- [macOS 系统音频权限和版本差异] → 首发限制为 macOS 14.2+，配置麦克风、屏幕和音频用途说明，并使用能力检测。
- [Universal 包体积接近翻倍] → 首发优先单一下载体验；体积影响显著时再拆分 x64 与 arm64 下载。
- [用户把采集程序误解为隐蔽监听] → 只允许显式开始，持续显示状态，并在桌面和 Web 提供停止与撤销。
- [回声或双声道重复导致问题识别错误] → 保持来源分离、添加时间戳，并在服务端做回声与重复段检测。
- [网络中断丢失音频] → 使用有界内存缓冲和序列确认；缓冲满后明确报告丢失，不写入无限本地缓存。
- [桌面安装和签名拖慢 MVP] → 先完成模拟采集程序和单平台开发包，再进入正式签名发布。
- [三个客户端状态难以一致] → 服务端仍是权威会话状态，桌面程序只发布采集事件，不维护业务答案状态。
- [传输或第三方转录泄露敏感音频] → 全程加密、最小权限、默认不保存，并在采集前展示数据流向。

## Migration Plan

1. 在 Web 原型中增加下载、连接和采集状态组件，使用模拟桌面设备验证流程。
2. 以 macOS 14.2+ 为最低版本，建立 universal 桌面伴随程序壳与签名策略。
3. 实现麦克风采集和本地音量诊断，再实现受支持的系统音频适配器。
4. 接入短期绑定、安全传输和服务端模拟接收。
5. 接入实时转录适配器，并把转录与问题事件同步到 Web 会话。
6. 完成权限、重连、撤销、版本兼容和隐私测试后再开放正式下载。

如果桌面采集程序不可用，Web 产品仍可使用手动输入、浏览器麦克风或截图回答，不阻塞核心流程。

## Open Questions

- MVP 是否必须分离麦克风和系统音频，还是允许先使用混合输入？
- 原始音频是否在任何付费或用户主动开启的场景下允许保存？
- 实时转录优先在服务端完成，还是需要评估系统本地转录作为隐私模式？
- 桌面程序是否需要自动更新，还是 MVP 先使用手动下载安装升级？
