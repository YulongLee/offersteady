## Context

当前 Web 原型已经包含三类 `ContextLibrarySource` 和会话级 `SessionContextSelection`，但资料管理页实际只管理知识库集合；简历与 JD 的上传/解析仍藏在准备页旧状态区域。`selectionValidity` 又把有效简历和 JD 作为开始面试的前置条件，因此空清单虽然在数据结构中可表达，却不能作为正式路径确认并启动。

实时工作台的截图与手动输入位于左侧资料栏，低置信度问题的“确认并回答”位于右侧 `SpeakerPanel`。平板和手机会把这两个侧栏转换为抽屉，用户必须先找到并打开抽屉才能回答，且底部还同时存在会话栏和移动导航，容易发生遮挡。

本设计复用现有资料来源、选择版本、截图任务和说话人状态，不引入新的解析或 AI 供应商。

## Goals / Non-Goals

**Goals:**

- 用三个直观分区管理简历、JD 和知识材料，同时保持统一的来源所有权与状态模型。
- 在单场准备页允许明确确认空选择或部分选择，不把资料完整度与能否开始面试混为一谈。
- 确保空选择不会触发未授权资料回退，并准确说明回答个性化程度。
- 建立一个由主回答区域持有的统一操作区，在桌面和手机都能处理确认、手动输入和截图。
- 保持响应式切换、抽屉开关和软键盘出现期间的操作状态与可访问性。

**Non-Goals:**

- 不支持同一场选择多份简历或多份 JD。
- 不增加资料自动推荐、自动全选、组合模板或团队共享。
- 不改变说话人检测、截图识别、回答生成和计费费率。
- 不在本变更中重构全部实时同步协议或实现原生手机应用。

## Decisions

### 1. 资料页使用类型分区，底层继续共享来源模型

资料页增加“简历”“职位 JD”“知识库”三个一级分区。简历和 JD 展示独立列表与类型相关的添加方式；知识库继续保留集合与多文档结构。三者统一映射到 `ContextLibrarySource`，不为 UI 分区复制一套会话来源模型。

备选方案是继续把所有内容放在“资料库”集合中并增加类型标签。该方案改动较少，但用户仍会误认为简历和 JD 是 RAG 知识文档，也难以提供粘贴 JD、替换简历等专属操作，因此不采用。

### 2. 将“选择完整性”和“个性化程度”拆开

`SessionContextSelection` 已允许 `resumeSourceId` 与 `jobDescriptionSourceId` 为 `null`，继续沿用该结构。保存空选择时仍写入 `confirmedAt` 和递增 `revision`，表示用户明确授权了一个空允许清单。

校验逻辑拆为两部分：

```ts
type SelectionIntegrity = "unconfirmed" | "valid" | "attention-required";
type ContextLevel = "none" | "resume-only" | "jd-only" | "knowledge-only" | "partial" | "personalized";

const canStart =
  selectionIntegrity === "valid" &&
  inputModeReady;
```

`valid` 只表示所有已选来源合法且清单已确认，不再表示简历和 JD 必须存在。`ContextLevel` 用于准备提示和回答标签，不作为启动门槛。通用数据用途复选框不参与启动；具体音频和截图操作仍单独授权或确认。

备选方案是增加“受限模式”开关。它会让用户先理解产品内部模式才能继续，并产生模式与实际选择不一致的问题；从已确认清单推导个性化程度更简单。

### 3. 每个资料分区都有显式空选择

简历和 JD 的单选组提供“不使用”；知识材料提供“全部取消”；准备页再提供“一键本场不使用资料”。最近资料只能作为未选中的建议，不能自动写入已确认清单。用户确认空清单时展示一次影响说明，但不使用恐吓式阻断。

### 4. 空清单作为真正的空服务端允许列表

服务端保存与校验选择 revision 后，把允许来源 ID 集合交给检索适配器。集合为空时跳过个人资料检索，不得回退到用户资料库、最近资料或默认知识库。回答 provenance 返回空 `usedSources`，UI 显示“未使用个人资料”，并继续禁止虚构经历。

备选方案是空选择时自动使用通用知识库。除非该知识库被明确标注为产品公共内容并与个人资料分开授权，否则会破坏来源可解释性，因此本次不采用。

### 5. 对话内确认与 `CompactQuestionBar` 分担明确动作

主区域使用 `CompactQuestionBar`，接收手动草稿、截图任务与命令回调；内容不清晰的问题继续属于同一会话状态，但由相关实时对话条目呈现：

```ts
interface LiveActionState {
  pendingQuestion: PendingQuestion | null;
  manualDraft: string;
  screenshotTask: ScreenshotTask | null;
}
```

紧凑输入栏位于回答区域之后，常驻提供“回答问题”和“截图回答”。`ConversationMonitor` 在低置信度文本旁显示“忽略 / 确认问题”，但角色由双声道固定为“我/面试官”且不提供角色纠正。实时页不提供资料侧栏或现场调整入口，答案历史通过回答区翻页。

备选方案是在主栏复制侧栏按钮。重复入口会造成两个组件分别维护草稿、截图弹窗和幂等状态，因此采用单一组件重排而不是复制。

### 6. 手机使用紧凑粘性操作条并避开既有底部控件

在手机断点下，实时对话、回答和紧凑输入栏依次堆叠；内容不清晰的问题保留在相关对话条目，输入栏展示“回答问题”和“截图回答”。主内容为会话栏、移动导航和 `env(safe-area-inset-bottom)` 增加等量底部留白。截图流程使用可滚动 sheet，软键盘出现时输入栏仍可滚动到可见区域。

所有主操作触点至少 44×44 CSS 像素。桌面端使用内容内操作区，不额外固定。

### 7. 操作状态位于页面会话层，不依赖响应式 DOM

`manualDraft`、`pendingQuestion`、`screenshotTask` 和命令幂等键保存在 `LivePage` 会话状态或专用 hook 中。CSS 负责重排同一个组件，不按断点渲染两份独立操作组件。这样旋转屏幕或调整窗口不会丢失状态或重复扣点。

## Risks / Trade-offs

- [允许空资料后回答可能更泛化] → 开始前和回答来源处明确展示个性化程度，绝不暗示使用了个人经历。
- [三个资料分区增加导航层级] → 使用同页标签或分段导航，并在空状态提供直接添加入口。
- [移动端固定区域占用屏幕] → 默认使用紧凑单行操作条，仅在需要确认或用户主动展开时增加高度。
- [多个底部固定组件产生遮挡] → 用统一 CSS 高度变量、safe-area 偏移和移动视口测试约束层叠关系。
- [从必选资料迁移到可空会改变旧准备度] → 将旧的资料就绪百分比迁移为“资料个性化程度”，不把缺失资料显示为错误。
- [空清单被误当成未保存] → 以 `confirmedAt` 和 revision 区分“已确认空清单”与“尚未确认”。

## Migration Plan

1. 先扩展选择校验与测试，使已确认空清单成为有效清单，同时保留所有权和资料状态校验。
2. 将资料页改为三个分区，把现有合成简历、JD 和知识来源迁移到对应列表；知识库集合结构保持不变。
3. 更新准备页的空选择、部分选择、影响提示和启动门槛，并迁移准备度文案。
4. 抽取 `LiveActionComposer`，把手动、截图和问题确认状态从左右侧栏移动到主区域。
5. 增加手机粘性操作条、safe-area 和软键盘布局，再移除侧栏中的唯一动作入口。
6. 运行空资料、部分资料、移动断点、截图失败、幂等扣点和可访问性回归测试。

若主操作区出现严重回归，可暂时回退为主栏内容内非粘性布局；不得回退到手机只能通过侧栏执行操作。资料迁移只改变呈现与选择校验，不删除已有来源。

## Open Questions

- 手机端常驻操作条是否同时保留“暂停/恢复”，还是继续由现有会话栏承担？原型可通过可用性测试决定最终合并方式。
- 资料管理页三个分区采用顶部标签还是左侧分段导航，需要结合手机原型宽度验证；行为规范不依赖具体视觉形式。
