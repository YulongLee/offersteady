## Context

当前原型已经把麦克风与系统音频作为独立来源传输，但 `add-speaker-role-detection` 又在来源之上引入 `candidate / interviewer / unknown` 角色推断、置信度和用户纠正。Web 因而出现“角色待确认”，用户需要在面试中处理模型内部不确定性。首发产品的实际约束更简单：用户选定的本地麦克风或耳机代表自己，电脑系统音频代表远端面试官。

回答任务目前只有生成、完成和失败等展示路径。用户发现问题重复、无关或不想回答时，缺少终止当前生成的控制；客户端即使隐藏内容，也无法可靠取消上游模型、阻止迟到分片或释放积分预留。

本变更跨越协议、Web、桌面伴随程序、回答编排和计费。实现前必须同时修订仍活跃的角色识别与音频 change，防止旧任务把三角色 UI 或“开始收音”再次带回产品。

## Goals / Non-Goals

**Goals:**

- 在受支持的双声道模式中，用来源直接得到“我/面试官”两个用户角色。
- 从实时页删除角色待确认、角色置信度和角色纠正操作。
- 保留问题边界、文本质量、回声去重和候选人误触发防护。
- 让用户可靠、幂等地终止正在排队或生成的回答。
- 让回答取消、迟到事件、重试和积分结算使用同一服务端最终状态。
- 把面向用户的主要启动操作统一为“开始面试”，同时保持明确授权和持续采集提示。

**Non-Goals:**

- 不识别面试官姓名、人数、性别、情绪或现实身份。
- 不在首发版本支持单个混合音轨上的自动角色推断或自动回答。
- 不移除手动提问、截图回答、暂停收音或结束面试。
- 不保证每个模型供应商都能立即停止计算或退还第三方成本；产品结算仍以自身任务状态为准。
- 不在本变更实现生产级语音或模型供应商，只保持可替换接口与原型状态可验证。

## Decisions

### 1. 用户角色由受信来源路由，而不是说话人推断

新路径在设备绑定和音频来源选择时建立会话级映射：

```text
selected microphone / headset input -> candidate -> “我”
computer system audio               -> interviewer -> “面试官”
```

服务端仍保留 `sourceId`、`sourceKind`、序列时间和可选匿名 speaker ID，用于排序、回声抑制和转录修订；这些字段不再决定用户可见角色。系统音频中即使检测到多个远端 speaker，UI 仍统一显示“面试官”。

备选方案是保留 `unknown` 并隐藏置信度，但这仍会留下第三角色和纠错流程，不能解决用户的核心干扰，因此不采用。

### 2. 来源异常以能力降级呈现，不伪造角色

新协议事件要求 `sourceKind` 为 `microphone | system`。缺少可信来源、只提供 `mixed` 或来自旧客户端的 `unknown` 事件进入兼容隔离：

- 不作为普通对话轮次显示第三角色；
- 不进入自动回答触发；
- 页面显示“系统音频不可区分，请使用手动提问或检查桌面程序”的状态；
- 已有对话、回答和音频权限状态不被清空。

备选方案是把所有异常片段默认归为面试官，但候选人回声会触发循环回答和扣费，风险高于漏掉问题，因此不采用。

### 3. 角色固定不等于取消问题质量门

自动回答管线调整为：

```text
独立声道 -> VAD / 回声与重复抑制 -> 转录与修订去重
         -> sourceKind === system -> 端点 + 完整性 + 问题意图
         -> QuestionConfirmed -> AnswerTask
```

`sourceKind === microphone` 永不自动触发回答。系统音频仍需最终转录、完整问题意图、稳定事件 ID 和去重条件。重叠或低文本置信度可以标记为“问题内容不清晰”并允许手动输入，但不再转换为“角色待确认”。

备选方案是系统音频中的任何停顿都立即回答，延迟较低但会把寒暄、铺垫和回声当成问题，因此不采用。

### 4. 回答任务使用单向最终状态与版本化取消

回答任务状态收敛为：

```ts
type AnswerTaskStatus =
  | "queued"
  | "generating"
  | "completed"
  | "failed"
  | "cancelled";

interface CancelAnswerCommand {
  interviewId: string;
  answerTaskId: string;
  expectedRevision: number;
  idempotencyKey: string;
}
```

只有 `queued` 和 `generating` 可转入 `cancelled`。`completed`、`failed` 和 `cancelled` 均为最终状态，不允许迟到事件把状态改回生成中。服务端先以条件更新裁决最终状态，再调用可替换模型适配器的 `abort(taskId)`；即使供应商不支持真正中断，事件发布层也丢弃取消后的分片。

备选方案是客户端使用 `AbortController` 后直接隐藏回答。它不能同步其他设备、释放服务端资源或处理供应商迟到结果，因此仅作为网络请求取消手段，不作为业务事实。

### 5. UI 控制绑定最新活动任务，不绑定当前历史页

`AnswerWorkspace` 接收当前历史页与独立的 `activeAnswerTask`。当用户查看最新且任务活动时，“终止回答”位于生成状态附近；用户翻看历史答案时，活动任务控制保留在回答栏固定状态区，避免按钮误指向正在阅读的旧答案。按钮使用文字可访问名称，点击后进入短暂 `cancelling` 客户端状态以防重复提交。

成功取消后主要建议区域显示“回答已终止”。已出现的临时分片不再作为完整建议、依据或复盘内容；原型可在当前会话中折叠展示“未完成内容”，持久化层默认只保留任务状态和必要审计元数据。

备选方案是把停止按钮放入底部手动提问栏，但它与具体生成任务距离较远，也容易被误解为停止面试，因此放在回答状态区域。

### 6. 任务状态和积分结算采用同一原子裁决

回答创建时沿用现有积分预留。状态从活动态成功变为 `cancelled` 的同一服务端操作发布一次预留释放事件；状态先变为 `completed` 时，终止命令返回 `not-cancellable`，按成功规则结算一次。幂等键和任务版本阻止重复释放、重复扣费或跨会话取消。

会员回答同样记录取消用量事件以便审计和质量评估，但金额与点数均为零。第三方供应商是否产生内部成本属于运营指标，不改变用户看到的取消结果。

### 7. “开始面试”是显式采集命令的产品文案

Web 准备页和已绑定桌面程序共享一个会话启动命令。来源未授权或测试失败时按钮不可直接开始采集；手动模式启动只改变面试会话状态，不触发音频权限。任一客户端成功启动后，其他客户端同步 `live/capturing` 状态并隐藏重复启动操作。

“暂停收音”“恢复收音”和持续采集指示保留技术准确性，因为它们明确控制音频而不是整场会话。仅把初始主操作“开始收音”改为“开始面试”。

备选方案是只改桌面按钮文字而不统一命令状态，可能造成 Web 与桌面分别启动两次，因此采用会话级幂等启动命令。

### 8. 先对齐活跃规范，再修改代码

Apply 的第一阶段必须修订：

- `add-speaker-role-detection`：删除用户可见 unknown、角色置信度、角色纠正和混合音频自动角色推断；保留回声、转录、问题边界、去重和评测。
- `add-desktop-audio-companion`：把启动场景和按钮改为“开始面试”，保留权限、暂停、停止与隐藏采集禁令。
- `add-web-interview-application`、`refine-interview-materials-and-mobile-actions` 和用户指引：移除待确认角色，但保留不清晰问题的手动提问路径。

协议可在一个兼容版本内继续解析 `unknown`，新客户端和 fixture 不再创建该角色。待所有受支持客户端升级后，再单独评估删除兼容枚举，避免本变更扩大为不可控迁移。

## Risks / Trade-offs

- [系统音频可能包含候选人回声] → 在固定角色映射前执行跨声道回声与重复抑制；冲突片段不触发回答。
- [共享电脑或混音设备破坏双声道假设] → 明确降级到手动模式，不显示虚假的第三角色或猜测结果。
- [取消请求与完成事件竞争] → 服务端使用任务版本和原子条件更新确定唯一最终状态。
- [供应商无法真正中断生成] → 适配器尽力取消，事件层忽略迟到分片并单独记录内部成本。
- [用户把终止回答误解为结束面试] → 使用完整文字“终止回答”，放在回答状态区，并在终止后保持明显的面试进行中状态。
- [新规范与多个未归档 change 冲突] → Apply 前先修订冲突 artifacts 并逐个严格校验，禁止只改代码。
- [隐藏 unknown 可能让少量转录不可见] → 展示来源级降级提示和手动输入入口，不把不可靠文本包装成确定角色。

## Migration Plan

1. 修订冲突的活跃 OpenSpec artifacts 和长期用户指引，并严格校验所有受影响 change。
2. 扩展协议和兼容层，增加回答取消命令/状态，同时停止新 fixture 生成 unknown 角色。
3. 更新服务端来源路由、问题触发、回答取消和计费预留释放的可替换适配器边界。
4. 更新 Web 双角色转录、降级提示、“终止回答”状态与跨设备同步。
5. 更新桌面伴随程序启动文案和幂等会话启动行为。
6. 使用合成数据运行单元、集成、AI eval 与浏览器响应式/无障碍检查。
7. 若取消或固定角色触发回归，可关闭自动音频回答并保留手动提问；不得回退到无提示的角色猜测或客户端假取消。

## Open Questions

- 无阻塞问题。首发明确只在可区分本地麦克风与系统音频的双声道模式启用自动回答；混合音频自动角色推断留待后续独立提案。
