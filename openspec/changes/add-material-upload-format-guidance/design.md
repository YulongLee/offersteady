## Context

当前资料页已经分成“简历”“职位 JD”“知识库”三个分区，并且在上传控件层面做了一部分格式限制，但这些信息主要藏在弹窗里的 `accept` 属性或局部提示中。用户如果只是刚进入资料页，往往看不到“到底支持上传什么格式”，尤其在简历和 JD 分区里，这个问题比知识库更明显。

这次变更不是要扩展上传能力，而是要把已有或计划中的文件支持范围说清楚，让用户在进入资料页时就能获得明确预期。结合当前产品原型，最需要解决的是三件事：

- 主界面进入即见，而不是“必须先点击上传才知道”
- 三个资料分区与对应弹窗文案保持一致
- JD 既支持文件又支持文本粘贴时，说明不要混淆

## Goals / Non-Goals

**Goals:**

- 在资料页主界面和空状态中明确展示支持上传 `PDF / DOCX / DOC / TXT / MD`
- 保持简历、JD、知识库三个分区与上传弹窗的格式说明一致
- 明确区分“支持文件格式”和“JD 也可直接粘贴文本”

**Non-Goals:**

- 不新增真实上传解析能力
- 不改动后端接口、收费规则、索引逻辑或资料选择流程
- 不重做资料页布局，只在现有结构中补充清晰说明

## Decisions

### 1. Put format guidance in the page-level content, not only in dialogs

格式说明必须出现在资料页主内容里，而不是只依赖弹窗。最合适的位置是：

- 每个资料分区标题下的说明文案
- 对应空状态文案
- 上传弹窗顶部说明

这样无论用户是否立即点击“添加”，都能先知道需要准备哪种文件。替代方案是只改弹窗文案，这样实现最省事，但依然解决不了“进入页面时看不到”的核心问题。

### 2. Standardize one canonical format list across all material types

这次统一采用一个规范格式列表：`PDF、DOCX、DOC、TXT、MD`。页面和弹窗都复用同一套文案表达，避免出现知识库支持 `MD` 但简历 / JD 文案遗漏，或者某些地方写英文扩展名、某些地方写中文别名导致理解不一致。

替代方案是每个分区写自己的版本，例如知识库强调 `Markdown`、简历只写 `PDF/DOCX`。如果后端真实限制不同，这种做法可以成立；但当前用户需求是“先明确知道能传什么”，因此先统一说明更能减少困惑。若未来某一类资料确实要收窄支持范围，应通过单独变更再次明确。

### 3. Separate file guidance from text-input guidance for JD

JD 分区除了文件上传，还支持直接粘贴文本。因此说明文案需要拆成两层：

- 文件上传支持：`PDF、DOCX、DOC、TXT、MD`
- 额外能力：也支持直接粘贴 JD 文本

替代方案是把两者写在一句混合文案里，例如“支持 PDF、DOCX、TXT、Markdown 或直接粘贴”。这种写法信息量够，但可读性一般，用户仍可能忽略文件范围或误解为粘贴只支持某些格式。

## Risks / Trade-offs

- [Risk] 当前原型代码里的 `accept` 属性如果没同步到 `DOC`，会让文案与实际选择器不一致
  Mitigation: apply 阶段需要同时检查并同步 `accept` 配置与说明文案

- [Risk] 页面文案增加后可能让资料页显得更密
  Mitigation: 复用现有说明区域和空状态，不新增复杂组件，只补一行高信息密度说明

- [Risk] 用户可能把“支持上传”理解成“上传后一定能高质量解析”
  Mitigation: 文案只说明支持的文件格式，不承诺解析成功率、索引速度或新增免费额度

## Migration Plan

1. 先在资料页三个分区确定统一格式说明文案。
2. 再同步简历、JD、知识库对应弹窗中的说明文字和 `accept` 范围。
3. 最后补前端测试，确认用户进入资料页即可看到说明，且不同位置文案一致。

回滚策略：如果文案布局影响阅读，可先回退主界面的说明位置，但不应回退到只有弹窗内可见的状态。

## Open Questions

- 简历、JD、知识库三类资料在真实业务落地时，是否都真的会支持完全相同的五种文件类型？
- 如果后端后续对某些格式做大小或解析限制，资料页是否还需要同时展示“推荐格式”与“可接受格式”两个层级？
