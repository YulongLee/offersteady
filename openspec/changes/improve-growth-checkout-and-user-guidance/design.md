## Context

现有原型已经具备公开首页、微信登录按钮、积分/会员页和三平台下载卡，但仍使用“产品原型”、偏抽象的价值表达、合成微信回调以及付款截图人工核验。Windows 构建目标已加入工程，实际签名、系统音频、配对和物理设备验证尚未全部完成。用户希望把这些入口升级为正式产品体验，同时建立统一图片资源和独立使用说明。

本变更影响营销内容、身份认证、支付安全、Windows 发布、静态资产治理与帮助内容。微信和支付宝的官方能力依赖主体资质、商户签约、HTTPS 回调和密钥管理，因此原型 UI 与真实可用状态必须明确分开。

## Goals / Non-Goals

**Goals:**

- 让用户在首屏理解“能帮助面试、可以免费开始、价格选择灵活”。
- 让“Windows 已支持”成为经过发布门槛验证的事实，而不是单纯文案开关。
- 用微信官方授权窗口完成登录、注册和账号绑定。
- 用微信支付/支付宝官方订单代替交易单号、付款截图与人工核验。
- 建立品牌、支付渠道、客服和客服群图片的统一资产目录与运营替换机制。
- 提供登录后独立使用说明，并从关键业务页面直达对应章节。

**Non-Goals:**

- 首页不承诺用户一定拿到 Offer，也不隐藏 AI 建议和真实经历边界。
- 不把个人微信/支付宝静态收款码作为正式支付方案。
- 不在当前范围支持 Windows ARM64、Linux 或移动端原生伴随程序。
- 不把支付商户密钥、微信登录密钥、证书或生产客服二维码硬编码到前端包。
- 使用说明不替代隐私政策、用户协议或支付服务协议。

## Decisions

### 1. 首页围绕结果、免费门槛与使用节奏组织

建议首屏文案：

> AI 面试助手，助你更从容地冲刺 Offer

> 实时理解面试问题，结合你的简历、岗位要求和个人资料，快速生成清晰回答思路。语音、手动输入和截图题都支持。

主 CTA 为“免费使用”，紧邻“新用户赠 200 点，无需先付费”。首屏下方使用三项直接价值：实时辅助、个性化回答、按自己的节奏付费。价格模块同时展示按次积分和 3/7/15/30 天会员，不再用“不是通用答案”等需要用户二次理解的否定句作为主卖点。

“辅助表达、不虚构经历”仍保留在可信度与使用边界区域，但不占据主标题。备选方案是完全删除边界文案，会提升短期冲击力，却可能制造结果保证和产品误解，因此不采用。

### 2. Windows 支持声明由发布准入状态计算

增加 `WindowsSupportReadiness`，只有以下条件全部通过时才返回 `supported`：

- Windows x64 安装包已代码签名且可下载；
- 安装、升级、卸载和自动更新验证通过；
- 微信登录/账号绑定、设备配对与重连通过；
- 麦克风授权和采集通过；
- 系统音频能力通过，或产品明确批准以受限模式发布并在所有入口披露；
- 至少完成规定的 Windows 10/11 物理设备矩阵验证；
- 当前协议版本与 Web 兼容。

当状态为 `supported` 时，首页、下载页和帮助页统一显示“支持 Windows 10/11 x64”。未满足门槛时不能通过营销文案配置强行覆盖。备选方案是现在直接写“已支持”，风险是用户下载后无法完成核心流程，因此不采用。

### 3. 微信登录使用官方授权弹层与服务端回调

桌面浏览器点击微信登录后打开登录弹层：显示由服务端创建的短期授权会话和微信官方二维码，或打开受控 popup 到微信授权地址。微信内浏览器使用官方 OAuth 重定向。回调页面校验一次性 `state`，服务端交换授权码，再通过受限 `postMessage` 或同源跳转通知原页面。

授权完成后：未绑定身份创建账号，已绑定身份登录；用户在设置中发起时绑定当前账号。微信身份属于其他账号时进入恢复/合并流程，不静默转移。弹层提供关闭、过期刷新、扫码中、成功、失败和提供方不可用状态。

备选方案是在客户端模拟成功或直接把二维码图片写死，无法证明身份且不可安全上线，因此只保留为开发环境 fixture。

### 4. 官方支付采用动态订单，不采集用户凭证

结算状态机为：`created → payment_pending → paid | failed | closed → refund_pending → refunded`。用户选择商品与渠道后，服务端从当前目录创建订单快照：

- 微信桌面端调用 Native 下单，返回短期 `code_url`，前端实时渲染订单专属二维码；微信内按资质使用 JSAPI。
- 支付宝调用电脑网站支付或当面付，跳转官方收银台或渲染订单专属二维码。
- 前端只显示订单号、金额、渠道、倒计时和支付状态，不要求填写交易单号或上传付款截图。

只有通过签名验证的异步通知，或通知缺失时服务端主动查单确认 `paid`，才能在同一幂等事务中激活积分/会员。浏览器返回页、二维码扫描、前端轮询结果或“支付完成”按钮都不是到账依据。二维码过期后关闭旧订单或由服务端创建新支付会话。

退款通过官方退款接口和异步状态完成；人工客服只能发起受权限控制的退款请求，不能直接把订单改成 paid。

### 5. 静态图片使用资产清单，动态支付码不入库

代码内资产目录：

```text
apps/web/public/assets/
├── brand/       # Logo、应用图标、favicon、分享图
├── payments/    # 微信支付/支付宝官方渠道图标，不含固定收款码
└── support/     # 客服微信、客服群、帮助入口二维码占位或公开配置资源
```

`assets.manifest.json` 记录稳定 ID、相对路径、用途、替代文本、尺寸、内容哈希、版本、公开状态和可选失效时间。组件通过资产 ID 或服务端公开配置引用，不散落路径字符串。生产客服/客服群二维码优先存放在受控对象存储并由配置覆盖，便于失效后立即替换；仓库只保留合成占位资源。

微信/支付宝订单二维码由官方下单结果动态渲染，绝不写入上述目录。这样既满足统一图片管理，也避免把长期静态收款码误当官方支付。

### 6. 使用说明采用独立路由和版本化内容

新增 `/app/guide` 与应用导航“使用说明”。内容按章节组织：快速开始、微信登录、资料库、面试准备、Windows/macOS 安装、现场回答、截图题、积分会员、兑换码、支付退款、数据隐私、常见问题和故障排查。兑换码章节只引导用户在积分页输入，并提醒不要在客服群或公开渠道发送完整兑换码。

中文内容保存在 `apps/web/content/guides/zh-CN/` 的 Markdown 文件中，构建时解析为白名单 HTML 或结构化内容；禁止原始脚本和不受信任 iframe。页面支持目录、搜索、上一步/下一步和稳定锚点。资料库、设备、收费、实时面试页通过“查看说明”跳到对应章节。

备选方案是只放一个长弹窗，移动端难以阅读、无法分享章节链接，也不利于版本管理，因此不采用。

## Risks / Trade-offs

- [“助你拿 Offer”被理解为结果保证] → 使用“助你更从容地冲刺”，并在可信区域明确 AI 建议和真实经历边界。
- [提前声称 Windows 支持导致信任受损] → 支持文案只能读取发布准入状态，不能由普通营销配置覆盖。
- [微信授权弹窗被浏览器拦截] → 在用户点击事件内打开 popup；失败时回退同页二维码或同窗口重定向。
- [支付回调伪造或重复] → 验签、金额/商户/订单校验、主动查单、幂等权益事务和回调重放测试。
- [支付二维码被缓存或截图复用] → 每单动态生成、短期过期、绑定订单金额，过期后服务端关闭。
- [客服群二维码过期] → 受控配置、资产版本和有效期检查；页面在失效时显示客服微信而非坏图。
- [图片版权或渠道品牌使用不合规] → 只使用官方品牌资源并记录来源与用途，不自行修改支付品牌标识。
- [帮助内容落后于产品] → 功能变更任务必须同时更新关联章节和内容版本。

## Migration Plan

1. 先引入资产目录、资产清单和使用说明路由，不改变支付。
2. 切换首页文案和“免费使用”CTA，运行转化与可访问性测试。
3. 完成 Windows 发布准入，状态通过后再统一切换“Windows 已支持”。
4. 在开发/沙箱环境接入微信授权弹层，关闭生产环境合成登录。
5. 接入微信支付与支付宝沙箱，完成验签、查单、退款和幂等测试。
6. 停止创建人工凭证订单；历史人工订单保持只读并继续原审核/退款处理。
7. 官方支付通过小流量验证后，删除新结算页中的交易单号、截图上传和人工核验入口。

回滚时可恢复旧首页文案或关闭支付渠道，但不能把已完成官方支付退回人工凭证流程，也不能撤销已合法发放的权益。

## Open Questions

- 首发微信登录使用开放平台网站应用扫码，还是公众号内 OAuth 为主入口？
- 微信支付首发采用 Native，支付宝采用电脑网站支付还是当面付？取决于商户签约范围。
- Windows “已支持”是否要求系统音频完整可用，还是允许明确标识的麦克风/手动输入受限版本？
- 客服群二维码是否长期稳定；若不稳定，是否改成客服自动拉群流程？
