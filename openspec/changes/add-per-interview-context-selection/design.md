## Context

OfferSteady 已有资料库、面试准备中心和回答来源展示的原型，但当前 `PreparationState.resources` 同时承担“资料库内容”和“本场面试内容”两种职责，实时工作台也默认展示全部就绪资料。这会在用户拥有多份简历、多个 JD 或不同领域知识材料时造成上下文污染。

本变更在资料库与面试会话之间增加显式的会话级允许清单。它只定义资料选择和来源追踪，不改变文档解析、向量检索或模型供应商。

## Goals / Non-Goals

**Goals:**

- 让用户明确选择每场面试使用哪份简历、哪份 JD 和哪些知识材料。
- 让不同面试的选择互不影响，并可在准备中心恢复和修改。
- 让进行中的变更只影响后续问题，保持历史回答可复现和可解释。
- 让服务端检索只能访问用户有权使用、已选择且已就绪的来源。
- 让每条回答显示实际使用的具体资料，而不是泛化的来源类型。
- 对停用、删除、失败和版本变化提供明确状态。

**Non-Goals:**

- 不自动判断“最佳资料组合”并绕过用户确认。
- 不在 MVP 中提供复杂标签规则、共享团队资料或批量权限策略。
- 不要求回答强行使用每一份已选资料；已选仅代表允许检索。
- 不因用户切换资料而自动重写历史回答。
- 不绑定具体向量数据库、检索算法或模型供应商。

## Decisions

### 1. 会话选择是资料库之上的允许清单

资料库继续保存用户可复用来源；每场面试保存独立的 `SessionContextSelection`：

```ts
interface SessionContextSelection {
  sessionId: string;
  resumeSourceId: string | null;
  jobDescriptionSourceId: string | null;
  knowledgeSourceIds: string[];
  revision: number;
  confirmedAt: string | null;
}
```

选择只引用资料 ID，不复制文档正文或索引。检索请求必须携带服务端已确认的选择版本，服务端以用户、会话、资料所有权和资料状态再次校验。

备选方案是在创建面试时复制资料。复制会带来版本漂移、重复存储和删除语义不清，因此不采用。

### 2. 零或一份简历、零或一份 JD、零到多份知识材料

资料选择允许零或一份已就绪简历、零或一份已就绪 JD，以及零到多份知识材料。简历和 JD 同时存在时可标为个性化；空清单或部分清单不阻止开始，但界面必须准确说明个性化程度，不能宣称使用了缺失资料。

MVP 不允许同一场同时选择多份简历或多份 JD，因为冲突事实和岗位要求难以解释。后续若有多语言简历需求，应增加明确的版本关系，而不是直接开放多选。

### 3. 默认建议不等于默认授权

新建面试时可以将最近使用的已就绪简历显示为“建议选择”，但不得自动选中，知识材料也默认不全选。用户必须在开始面试前确认清单，包括明确确认空清单；“全选知识材料”是显式操作，并显示数量和潜在噪声提示。

备选方案是自动选中所有资料。它操作更少，但违背最小必要处理原则，也容易引入无关内容，因此不采用。

### 4. 准备中心使用分组选择器

“本场使用的资料”卡片按简历、JD、知识材料分组，并显示类型、名称、更新时间、解析状态和选择状态。知识材料支持搜索、逐项多选、全不选和已选数量，不在首版加入复杂标签筛选。

```text
┌───────────────────────────────────────────────────────┐
│ 本场使用的资料                              已选择 5 项 │
├───────────────────────────────────────────────────────┤
│ 简历（单选）                                           │
│ ● 高级前端工程师简历 · 2 天前 · 已就绪                 │
│ ○ 产品工程师简历     · 1 月前 · 已就绪                 │
├───────────────────────────────────────────────────────┤
│ JD（单选）                                             │
│ ● 示例科技高级前端 JD · 今天 · 已就绪                  │
├───────────────────────────────────────────────────────┤
│ 知识材料（多选）                  [搜索] [全部取消]     │
│ ☑ 前端性能治理  ☑ 微前端复盘  ☑ 系统设计清单           │
│ ☐ 产品方法论    ⛔ 旧版笔记 · 解析失败                 │
└───────────────────────────────────────────────────────┘
```

准备进度使用“选择有效且已确认”判断，而不是资料库总数。

### 5. 面试中固定使用开始前确认版本

实时工作台不再显示资料侧栏或“调整资料”入口，持续使用开始前最后确认的选择 `revision`。每条回答仍保存实际使用的来源和 revision；若用户需要改变资料，须离开当前实时会话并回到准备流程重新确认。

备选方案是立即影响正在生成的回答。该方案会产生不可复现的混合答案，因此不采用。

### 6. 区分允许来源与实际使用来源

已选资料是检索允许范围，并不表示每次回答都实际使用。回答事件返回：

```ts
interface AnswerSourceRef {
  sourceId: string;
  sourceVersion: string;
  displayName: string;
  kind: "resume" | "jd" | "knowledge";
}

interface AnswerProvenance {
  selectionRevision: number;
  usedSources: AnswerSourceRef[];
}
```

回答区默认显示实际使用的来源名称；展开后可查看类型和版本。未命中的已选材料不显示为回答依据，避免夸大来源。

### 7. 服务端强制执行资料范围

浏览器传来的 source ID 不能直接作为授权依据。可信服务必须验证资料归属、会话归属、选择 revision 和 `ready` 状态，再将允许 ID 传给可替换检索适配器。前端不得持有检索或模型供应商密钥。

### 8. 资料失效不会被静默替换

已选来源被停用、删除、解析失败或更新成新版本时，当前选择进入 `attention-required`。删除来源立即从后续检索排除；版本更新保留旧版本引用用于历史来源说明，但新问题只有在用户确认新版本后才能使用它。

## Risks / Trade-offs

- [用户每次都要选择资料，流程变长] → 提供最近简历建议和清晰的已选摘要，但保留显式确认。
- [选择太多材料仍然污染检索] → 显示数量提示，默认不全选，并继续执行相关性阈值。
- [面试中切换导致历史答案难以解释] → 选择版本化，每条回答记录 selection revision 和实际来源版本。
- [资料删除后历史来源断裂] → 删除正文与索引，但历史回答保留最小化的来源名称、类型和已删除标识，不保留正文。
- [前端篡改 source ID 越权读取] → 服务端按用户、会话、选择和状态四层校验。
- [已有原型只有一份知识库聚合项] → fixtures 拆成多个独立来源，再增加会话选择模型，不复用聚合项作为真实来源 ID。

## Migration Plan

1. 扩展 Web 领域模型，将资料库来源和会话选择拆开。
2. 使用多个合成来源实现准备中心选择器和实时工作台摘要。
3. 增加选择有效性、版本变化、删除与历史来源测试。
4. 定义并接入服务端选择保存、读取和 revision API。
5. 修改检索适配器，只接收服务端校验后的允许来源 ID。
6. 为已有面试创建空的未确认选择，要求用户进入准备中心确认，不自动全选历史资料。

若后端选择 API 尚未就绪，前端可保留 fixture 选择器，但不得把本地选择描述为已同步或已用于真实检索。

## Open Questions

- 简历更新后，进行中的面试是否允许继续使用旧版本直到结束？
- 是否需要让用户保存“资料组合模板”，还是等单场选择验证后再加入？
- 回答来源是否展示具体片段，还是 MVP 只展示资料名称和版本？
