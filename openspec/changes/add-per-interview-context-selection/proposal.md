## Why

用户的资料库可能同时包含多份简历、不同岗位 JD 和多类知识材料，如果系统默认把所有资料都用于每场面试，回答容易被无关或冲突内容污染。每场面试需要一份由用户明确确认的资料选择，才能让回答相关、可解释且符合最小数据使用原则。

## What Changes

- 在面试准备中心新增“本场使用的资料”选择器，而不是只展示资料库整体是否就绪。
- 允许用户为每场面试选择一份简历、一份目标 JD，以及零到多份知识库材料。
- 显示每份资料的名称、类型、处理状态、更新时间和是否已选；未就绪、停用或已删除资料不可用于新回答。
- 默认不自动勾选全部知识材料；新建面试可建议最近使用的简历，但必须由用户确认本场选择。
- 保存会话级资料清单，确保不同面试拥有互不影响的选择。
- 允许在面试进行中打开轻量选择面板；变更只影响后续问题，不重写已经生成的回答。
- 每条回答记录实际参与检索或生成的资料来源，不能仅显示笼统的“知识库”。
- 当已选资料被停用、删除、解析失败或版本发生变化时，显示失效状态并要求用户处理，不静默替换。
- MVP 不提供复杂自动规则、智能资料组合或团队共享资料权限。

## Capabilities

### New Capabilities

- `per-interview-context-selection`: 为每场面试选择、确认、变更并追踪简历、JD 与知识材料的使用范围。

### Modified Capabilities

无。当前相关能力仍处于未归档变更中；本变更以独立能力补充会话级资料选择，不修改已归档规范。

## Impact

- `apps/web` 的准备中心、资料库、实时工作台和回答来源区域需要增加选择与失效状态。
- 前端领域模型需要区分“资料库中的资料”“本场已选择资料”和“某条回答实际使用资料”。
- 后端需要保存带用户与会话作用域的资料选择，并在检索请求中只允许已选择且已就绪的来源。
- 检索与回答管线必须接收允许的来源 ID 集合，并在输出中返回实际使用的来源 ID 和版本。
- 简历、JD 与知识材料均属于敏感数据；按场选择减少不必要处理，但仍需访问控制、删除和审计边界。
