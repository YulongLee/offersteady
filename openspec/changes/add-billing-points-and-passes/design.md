## Context

OfferSteady 当前只有使用合成数据的 Web 原型，尚无真实账号、商品目录、钱包、订单或支付基础设施。实时回答和截图回答未来会产生模型、语音、存储和人工支持成本，因此计费必须在真实 AI 接入前形成清晰边界。

用户希望同时提供按积分使用和按天无限使用，并用微信或支付宝付款截图完成早期收费。本设计把截图支付定义为临时人工核销方案：截图只能作为线索，权益必须在后台与商户收款记录核对后发放。正式规模化应切换到微信支付/支付宝商户 API 的服务端下单、支付通知、主动查单和对账流程。

## Goals / Non-Goals

**Goals:**

- 给新用户一次性 200 点试用额度并提供完整积分流水。
- 同时支持按次扣点和 3/7/15/30 天会员权益。
- 在面试开始前和每次计费操作前清楚展示价格、余额与权益。
- 对回答、截图、失败、重试、退款和重复请求建立幂等计费语义。
- 为微信/支付宝人工凭证审核提供安全、可审计的 MVP 流程。
- 让商品、点数、价格和毛利约束由服务端配置。
- 在收费页提供订单记录、退款说明和售后客服微信入口。

**Non-Goals:**

- 不承诺任何定价在缺少真实成本数据时必然盈利。
- 不使用付款截图或 OCR 结果自动确认到账。
- 不允许前端直接修改积分、会员或订单状态。
- 不在 MVP 中实现自动续费、分销、优惠券、企业套餐或复杂促销。
- 不将“无限使用”解释为允许账号共享、无限并发、自动化滥用或绕过安全限制。
- 不在客户端保存微信支付、支付宝或模型供应商密钥。

## Decisions

### 1. 使用服务端商品目录，不在前端写死价格

商品分为 `points_pack` 和 `time_pass`。前端只渲染服务端返回的已上架商品，订单保存购买时的商品快照，避免后续调价改变历史订单。

试运营建议值如下，属于待验证配置而不是最终承诺：

| 商品 | 建议售价 | 权益 |
| --- | ---: | --- |
| 3 天会员 | ¥69.90 | 72 小时内正常个人使用回答与截图不扣点 |
| 7 天会员 | ¥129.90 | 168 小时内正常个人使用回答与截图不扣点 |
| 15 天会员 | ¥219.90 | 360 小时内正常个人使用回答与截图不扣点 |
| 30 天会员 | ¥329.90 | 720 小时内正常个人使用回答与截图不扣点 |
| 300 点 | ¥39.90 | 300 个永久积分 |
| 800 点 | ¥89.90 | 800 个永久积分 |
| 2000 点 | ¥199.90 | 2000 个永久积分 |

建议初始用量费率：普通回答 5 点，截图回答 15 点。语音转写本身不单独扣点，成功生成回答时结算；后续若真实成本证明不可持续，再通过新目录版本调整，并提前展示给用户。

备选方案是在组件中写死价格。它实现快，但无法安全调价、下架或做历史审计，因此不采用。

### 2. 会员优先，积分作为回退

服务端决定每次操作的计费来源：

1. 有有效会员且符合正常使用规则：允许操作，记录 0 点会员用量。
2. 没有有效会员：按当前费率从积分钱包预留。
3. 余额不足：不创建新的 AI 任务，返回可恢复的余额不足状态。

会员到期后自动回到积分模式，原有积分不清零。重复购买会员时，新权益从 `max(当前时间, 当前会员结束时间)` 开始顺延。

### 3. “无限使用”包含透明的正常使用边界

会员页面显示“会员期内回答和截图不扣积分”，同时说明仅限账号本人、单场主要会话和合理请求频率。安全限流、最大并发、模型上下文上限和违反产品规则的请求仍然生效，但不得用隐藏限额把会员悄悄退化为按次套餐。

服务端记录会员用量和估算成本。当某 SKU 的实际毛利低于配置阈值时，只告警和停止新销售，不中途削减已购买用户的权益。

### 4. 积分钱包采用不可变流水

钱包余额使用整数点数，不使用浮点。余额由已入账流水汇总并缓存，流水类型至少包括：

- `welcome_grant`：一次性 200 点。
- `purchase_credit`：积分包到账。
- `usage_reserve`：创建任务前预留。
- `usage_settle`：任务成功后结算。
- `usage_release`：失败或取消后释放。
- `refund_debit`：退款时撤销未使用购买积分或按规则处理。
- `support_adjustment`：双人审核的人工调整。
- `redemption_credit`：一次性兑换码原子入账。
- `redemption_reversal`：授权审核后关联原兑换记录的负向冲正。

200 点只向完成有效账号验证且从未领取过的用户发放一次，不可转让或提现。反滥用规则不得收集与目的无关的敏感数据。

兑换码点数同样进入该钱包。服务端在一个事务中锁定兑换码、追加 `redemption_credit` 并更新余额；同一码与同一兑换 reference 具有唯一约束。客户端只提交明文码与幂等键，不提交可信点数；完整兑换码不得进入日志或持久化浏览器存储。

### 5. 用量采用预留—结算，而不是点击即扣费

每个回答或截图操作生成全局唯一 `usageId`。服务端在创建 AI 任务前检查权益并预留点数；只有任务成功交付可用回答时才结算。同一 `usageId` 的重试、客户端重复提交或消息重复投递返回同一计费结果。

截图回答按 15 点作为完整操作计费，不再额外叠加 5 点普通回答。用户取消上传、图片无法识别、生成失败或服务端超时均不结算。

### 6. 收费页面与面试现场保持价格透明

收费页路由建议为 `/app/billing`，页面结构：

```text
┌─────────────────────────────────────────────────────────────┐
│ 我的权益：200 点                     [订单记录] [联系客服]  │
├─────────────────────────────────────────────────────────────┤
│ 按时间 · 会员期内不扣点                                   │
│ [3天 ¥69.9] [7天 ¥129.9] [15天 ¥219.9] [30天 ¥329.9]     │
├─────────────────────────────────────────────────────────────┤
│ 按次数 · 用多少付多少                                     │
│ 普通回答 5点 · 截图回答 15点                              │
│ [300点 ¥19.9] [800点 ¥39.9] [2000点 ¥79.9]                │
├─────────────────────────────────────────────────────────────┤
│ 支付方式：微信 / 支付宝                                   │
│ 创建订单 → 扫码付款 → 上传凭证 → 人工核对 → 权益到账      │
│ 售后客服：[微信二维码] [复制微信号]                       │
└─────────────────────────────────────────────────────────────┘
```

实时工作台顶部显示“会员剩余 6 天”或“余额 185 点”；截图按钮显示 `15 点`，普通问题显示 `5 点`。价格变化只影响尚未创建的用量，已预留操作使用预留时的费率快照。

### 7. 人工付款使用订单状态机

用户必须先选择商品并创建唯一订单，再查看配置的商户收款码。状态机：

```text
awaiting_payment
  -> proof_submitted
  -> under_review
  -> paid | rejected | expired
paid -> refund_pending -> refunded | refund_rejected
```

凭证包含支付渠道、付款截图、用户填写的交易参考号和付款时间。后台审核者必须核对商户账单中的金额、时间、交易号和订单号；截图 OCR 只能辅助输入。上传文件计算摘要用于重复检测，原图加密保存并按期限删除。

订单入账使用幂等事务：同一订单只能发放一次权益。客服可以查看状态和提交备注，但余额调整、审核通过和退款使用独立权限并记录审计日志。

### 8. 官方商户 API 是目标支付路径

支付适配器提供 `createPayment`、`queryPayment`、`handleNotification`、`refund` 和 `reconcile` 接口。官方支付回调或查单结果是自动入账依据；浏览器的“支付完成”按钮不是依据。

微信支付官方流程要求商户后端下单，并通过支付回调或主动查单确认订单状态。人工截图方案只用于小规模验证，在获得商户资质和公网回调能力后应迁移到 Native/JSAPI 等正式流程。

### 9. 售后入口由安全配置提供

客服微信号、二维码资源、服务时间和退款说明来自服务端公开配置。页面提供二维码、复制微信号和携带订单号的咨询提示，不把个人微信号写入源码。二维码只能指向经过运营确认的客服账号，并允许快速撤换。

### 10. 盈利使用单位经济约束，而不是主观保证

每种操作记录脱敏的模型、语音、存储和支付成本估算，不记录问题正文。商品发布前计算预期毛利；建议试运营毛利目标不低于 60%，并单独计入退款率、支付手续费和人工审核时间。

会员 SKU 使用按用户日、回答数、截图数和成本分位数监控。若无法满足目标，应调整新订单价格、模型路由或会员售卖策略，而不是影响已经支付的权益。

## Risks / Trade-offs

- [人工付款截图可伪造] → 截图不自动到账，必须核对商户账单并做重复检测。
- [人工审核成本侵蚀利润] → 限制为早期 MVP，记录审核耗时并优先迁移官方商户 API。
- [无限会员被重度使用导致亏损] → 透明正常使用边界、成本监控、SKU 毛利告警和停售机制。
- [200 点被批量注册套利] → 完成有效账号验证后只发一次，并使用最小化的反滥用信号。
- [重复请求造成多扣点] → 全局 usage ID、预留—结算和数据库唯一约束。
- [价格调整引发争议] → 服务端版本化目录、订单和用量保存价格快照、扣费前展示。
- [退款后积分已被使用] → 发布前明确退款规则，退款流程校验剩余权益并人工处理异常。
- [客服微信二维码泄露或被替换] → 使用受控配置、资源完整性检查、运营审批和审计。
- [付款截图包含敏感信息] → 限制格式与大小、加密存储、权限隔离、日志脱敏和到期删除。

## Migration Plan

1. 先用合成数据实现收费页、余额、套餐、扣费提示和订单状态原型。
2. 建立服务端商品目录、权益、钱包流水、用量预留和幂等测试。
3. 接入人工订单、商户码配置、凭证上传、审核后台和客服入口。
4. 以内部沙盒订单验证发放、重复审核、失败释放、退款和审计。
5. 小规模试运营并采集单位经济数据，再确认最终费率和售价。
6. 获得商户接入条件后迁移微信支付/支付宝官方 API；保留人工流程仅作异常补单。

若计费服务异常，系统停止创建新的付费任务并明确提示，不允许前端自行放行或扣除未知余额。已有回答和订单记录保持只读可访问。

## Open Questions

- 试运营价格是否采用本文建议值，还是先只展示套餐结构等待成本测算？
- 200 点是否永久有效，还是设置明确且提前展示的试用期限？
- 退款是否支持未开始会员全额退、已开始会员不退，积分包是否按未使用部分处理？
- 正式支付首选微信 Native、微信 JSAPI、支付宝电脑网站支付还是聚合商户服务？
- 客服服务时间和承诺响应时效是多少？
