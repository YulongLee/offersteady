## Context

OfferSteady 的 MVP 已规划资料上传、知识库、实时回答和截图回答。新的前端需要同时运行在电脑和手机上，但两个设备在面试中的职责不同：电脑适合管理资料、阅读完整回答和回看历史；手机更适合快速控制会话、查看当前提示和拍照提交题目。

本设计先定义前端页面和同步模型。它不假定两个设备必须同时在线，也不要求首版拥有原生 App。

## Goals / Non-Goals

**Goals:**

- 设计一个低干扰、桌面优先的面试主工作区。
- 设计一个可单手操作的手机伴随页面。
- 让两个设备看到一致的会话、问题和回答状态。
- 支持快速配对、设备角色切换、断线恢复和明确的隐私状态。
- 保持上传简历、JD、知识库、实时回答和截图回答入口清晰可达。

**Non-Goals:**

- 不在本变更中实现 iOS、Android 或桌面原生应用。
- 不要求手机和电脑显示完全相同的信息密度。
- 不支持多个设备同时采集音频。
- 不支持隐蔽录音、后台持续监听或绕过系统权限。
- 不在前端设计阶段确定具体实时通信供应商。

## Decisions

### 1. 电脑是主工作区，手机是伴随控制器

电脑端承担资料管理、完整回答、来源展开和问题历史；手机端突出当前回答、会话控制、拍照问答和连接状态。两个端都能查看核心信息，但不会机械复制相同布局。

备选方案是完全镜像界面。该方案实现简单，但会让手机界面信息过载，并浪费拍照和单手操作优势，因此不采用。

### 2. 页面结构与视觉方向

第一版采用沉静的深色专注界面：深蓝黑背景、低对比表面、青绿色品牌强调色。颜色只辅助表达状态，所有状态同时使用文字或图标说明。

响应式规则：

- `>= 1200px`：三栏桌面布局。
- `768–1199px`：资料抽屉 + 回答主栏 + 可折叠历史栏。
- `< 768px`：单栏手机布局，底部导航与会话操作保持可达。

### 3. 实时回答区域是唯一视觉主角

中心区域按“当前问题 → 回答提纲 → 完整回答 → 来源”组织。生成过程中保留已接收的问题并流式追加内容；历史回答降级到右栏或手机历史页，避免当前面试被旧内容淹没。

### 4. 服务端维护权威会话状态

客户端发送带唯一命令 ID 的意图，服务端校验后生成递增版本的会话事件。所有设备只根据已确认事件更新共享状态，从而避免两个设备同时操作造成分叉。

备选方案是浏览器之间点对点同步。该方案会增加 NAT、离线和安全处理复杂度，也不利于恢复，因此不作为 MVP。

### 5. 使用短期配对码连接设备

同一账号可以直接选择进行中的会话；也可以由电脑展示短期二维码和 6 位配对码，让已登录手机加入。配对码一次使用、短期有效，并可由主设备撤销。

### 6. 每场会话只有一个主要输入设备

主要输入设备负责实时音频或问题采集，其他设备只能查看状态、提交显式手动问题或请求切换角色。角色切换需要确认，并在完成前停止旧设备采集，以避免重复问题。

### 7. 增量同步与恢复

在线时优先使用双向实时连接传递命令和增量事件；断线后客户端展示离线状态，只缓存安全且幂等的显式操作。重连时客户端提交最后确认的版本号，服务端返回缺失事件或完整快照。

具体实时通信库由实现阶段选型，但接口必须可替换。

## Risks / Trade-offs

- [跨设备状态复杂度拖慢原型验证] → 第一阶段只同步会话、当前问题、回答和设备状态，资料编辑仍由电脑端完成。
- [两个设备重复提交问题] → 使用命令 ID、服务端幂等处理和单一主要输入设备。
- [手机屏幕信息过载] → 默认展示回答提纲，完整回答和来源按需展开。
- [网络抖动导致用户误判采集状态] → 始终显示连接状态和主要输入设备；断线时停止宣称“正在同步”。
- [二维码或配对码泄露] → 使用短期、一次性凭证，并要求已登录设备确认加入。
- [敏感回答在锁屏或通知中暴露] → MVP 不把完整问题或回答写入系统通知。

## Migration Plan

1. 先使用合成数据实现桌面和手机静态页面，完成响应式视觉验收。
2. 接入单设备本地会话状态，验证页面交互。
3. 增加服务端会话快照、事件版本和设备列表。
4. 接入配对、实时同步、重连和主要输入设备切换。
5. 最后连接 `define-interview-assistant-mvp` 中的真实资料与回答管线。

如果同步实现未达到稳定标准，可以保留响应式单设备界面并关闭设备配对入口，不影响核心面试工作区。
