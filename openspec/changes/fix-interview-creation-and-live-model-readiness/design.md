## Overview

本变更聚焦当前最核心的可用性链路：让“新建面试、最近面试维护、开始面试、手动快答”这四段行为都落在真实后端状态和真实模型运行时上，同时不改变现有产品原型布局。

## Goals

- 修复新建面试失败或创建后不可继续的问题
- 将最近面试主列表收敛到 5 条并支持删除维护
- 移除 Web 手动模式开始面试时不必要的额外令牌前置条件
- 确保快答/手动问题输入走后端真实 Chat Service 与 `.env` 模型配置

## Non-Goals

- 不改首页、准备页、实时页的主视觉布局
- 不改积分、会员、截图回答、实时语音、微信登录或支付流程
- 不在本次变更中扩展新的模型供应商，只拉通当前已接入运行时

## Design Decisions

### 1. 新建面试以后端会话创建结果为准

首页“新建面试”按钮不再先乐观插入前端假数据。前端提交创建请求后，只有在后端返回真实 Session 记录后才更新最近面试列表并跳转准备页；失败时展示真实错误。

这样可以避免：

- 首页出现“看起来创建成功、实际后端不存在”的幽灵面试
- 后续准备页、开始面试、快答都绑定到不存在的 session id

### 2. 最近面试列表只维护一个五条主队列

最近面试区域继续作为首页主入口，但只保留最新 5 场。实现上由后端聚合接口或会话查询层统一返回排序后的前 5 条，前端不自行做多份排序规则。

删除后行为：

- 服务端确认成功后从列表移除
- 若存在第 6 条及之后记录，则自动补位到最近面试主列表
- 失败时保留原记录并显示错误

### 3. “开始面试”只负责启动 session，不要求手动模式额外令牌

当前 Web 手动输入模式的核心需求是：用户确认资料后即可进入实时面试并提问。这里不再暴露或要求额外 token、publisher credential、audio binding token 等前置条件。

边界：

- 音频发布、桌面伴随程序、实时语音仍可在各自链路内部使用短时令牌
- 但这些令牌不应成为“Web 手动模式开始面试”的用户可见门槛
- Session Start 成功后，手动快答即可直接工作

### 4. 手动快答统一走真实后端模型运行时

前端 `快答` 与手动问题提交只负责：

- 发送 `sessionId`
- 发送用户问题
- 呈现后端返回的 streaming / completed / failed 状态

后端负责：

- 验证会话已启动
- 加载当前 `.env` 模型配置
- 调用现有 Chat Service / LLM Gateway
- 返回真实 usage、错误和回答内容

前端运行路径中不得再回退到 synthetic answer、fixture answer 或本地拼接回答。

## Architecture Notes

### Frontend

- 修正新建面试动作的成功/失败状态机
- 最近面试列表支持服务端删除后的补位刷新
- 准备页开始面试动作必须等待 `session start` 成功
- 实时页快答错误文案直接映射后端安全错误

### Backend

- Session create/start API 作为唯一事实源
- Dashboard / app-shell 聚合状态返回最近 5 场面试
- Live Answer 调用必须命中真实 Chat Service，而不是测试替身
- 若模型配置缺失或供应商失败，返回结构化错误，供前端展示

## Risks and Mitigations

- 风险：当前部分页面仍可能保留旧的本地 fallback 路径
  - 缓解：本次实现时重点检查新建面试、开始面试、快答三条入口的 adapter 与 runtime 分支

- 风险：最近面试只显示 5 条后，用户误以为更早记录丢失
  - 缓解：文案明确这是“最近面试”主列表；删除是显式操作，超出 5 条只是移出首页主列表而非静默删除

- 风险：真实模型错误直接透出会造成信息不一致或暴露内部细节
  - 缓解：前端展示后端返回的安全错误摘要，不展示密钥、供应商内部堆栈或敏感配置

## Verification

- 新建面试成功/失败的前后端联调测试
- 最近面试超过 5 条时的排序、补位、删除回归测试
- 准备页“开始面试”不再被额外 token 门槛阻塞
- 手动问题输入/快答命中真实后端模型链路的集成测试
- `.env` 模型未配置、配置错误、供应商失败时的错误展示测试
