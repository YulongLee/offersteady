## Context

当前仓库已经同时存在原型层、后端基础工程、统一文档服务和若干体验优化变更。随着工程接入逐步增加，`apps/web` 这套被用户反复确认过的原型页面，已经不只是“演示界面”，而是整个产品决策的事实基线。一旦本地原型无法稳定启动、页面被顺手改动，后续无论是评审、联调还是继续做 Spec，都容易建立在错误认知之上。

这次变更的目标不是新增能力，而是先恢复一套可信的本地原型，再以该原型为中心完成功能 review，并把“哪些体验是受保护基线”记录下来，约束后续开发。

## Goals / Non-Goals

**Goals:**

- 恢复本地 Web 原型的可运行性和关键页面可用性
- 以“已批准原型”为准，审查核心页面路径、文案和交互是否仍然一致
- 明确原型层与工程接入层的边界，降低后续再次误改原型的概率
- 输出一份后续可复用的恢复与巡检基线，便于继续 apply 前先对齐状态

**Non-Goals:**

- 不在本次变更中新增业务功能、重写页面结构或调整商业策略
- 不修改已批准产品需求，只识别问题并恢复到既有基线
- 不引入新的服务端能力、数据库设计或生产级部署改造
- 不把这次 review 结论自动视为新需求；如需改产品行为，必须另开变更

## Decisions

### 1. 采用“基线恢复优先”而不是“边改边优化”

先恢复到用户已确认的页面结构、流程和文案，再做功能 review。这样可以避免在定位原型损坏原因时，把“修复”和“设计优化”混在一起。

备选方案：

- 直接在当前代码上继续优化页面：速度看似更快，但会继续放大偏差
- 完全回退整个仓库：风险过大，容易误伤后端和文档等已确认工作

### 2. 以用户路径为单位做功能巡检

review 按完整路径执行：首页 → 登录/注册 → 资料库 → 面试准备 → 实时面试 → 积分/充值 → 下载入口 → 使用说明，而不是按组件零散检查。因为当前问题本质上影响的是产品可验证性，不只是单个组件是否报错。

备选方案：

- 仅按代码模块 review：更适合工程重构，不适合验证产品体验完整性

### 3. 通过兼容层和边界说明保护原型，不用顺手改原型适配新工程

当后端契约、上传能力或运行时配置与原型存在张力时，优先在 adapter、fixture 或兼容层处理，避免再次直接侵入原型页面结构。这样更符合当前“先验证产品，再推进正式工程”的阶段目标。

备选方案：

- 让原型直接跟随最新后端形态调整：会降低原型稳定性，并使用户确认过的页面不断漂移

### 4. 审查结论以“恢复项 / 待决策项”分层输出

本次 review 只把内容分为两类：一类是必须立即恢复的原型基线问题，另一类是观察到但需要后续单独决策的产品问题。这样可以避免 review 文档再次演变成隐性需求池。

## Risks / Trade-offs

- [风险] 当前“已批准原型”可能分散在多个历史变更与页面实现中，不易一次性对齐 → [缓解] 以最近已 apply 的体验变更、`docs/user-journey.md` 和现有 Web 页面为共同基准，优先锁定关键路径
- [风险] 恢复过程中可能再次误伤已接入的工程基础设施 → [缓解] 优先恢复 UI 表现和适配层，不随意删改后端基础工程
- [风险] 功能 review 容易滑向重新定义产品 → [缓解] 明确区分“恢复原型”与“新增需求”，新增行为必须另起 OpenSpec 变更
- [风险] 本地可运行不代表所有原型细节都与用户记忆一致 → [缓解] 输出审查结论时标记已恢复范围和仍待用户确认的差异点

## Migration Plan

1. 识别本地原型当前启动、路由和关键页面异常点
2. 对照已批准流程恢复首页、资料页、准备页和实时页等核心路径
3. 使用本地运行与页面巡检验证关键路径恢复情况
4. 记录受保护原型基线和功能审查结论，为后续实现提供约束

如果恢复过程发现当前变更无法在不改变需求的前提下解决问题，则停止扩展实现，并把冲突点记录为后续待决策项。

## Open Questions

- 当前“本地原型崩掉”是启动失败、编译失败，还是页面流程异常，需要在实施时先定位
- 是否需要补一份更显式的“受保护页面清单”，避免后续成员只读代码难以判断哪些页面不能动
