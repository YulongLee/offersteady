## Context

OfferSteady 已有整数积分钱包、不可变流水、新用户赠送、积分包购买、会员优先和 AI 用量预留。兑换码应成为钱包的第三种可信入账来源，而不是在前端直接增加余额。兑换码本质上是可转移的 bearer secret：任何得到明文的人都可能在失效前兑换，因此生成、存储、传输、日志和并发处理都需要明确边界。

首发用途是内测邀请、运营活动和售后补偿。产品仍处于原型阶段，本设计提供可替换的存储、密码学和限流接口，不提前绑定具体数据库或云服务。

## Goals / Non-Goals

**Goals:**

- 在积分页提供简单、清晰且可访问的积分兑换入口。
- 保证一次性码只能成功兑换一次，重试和并发不会重复入账。
- 把兑换积分写入现有钱包和不可变流水，不创建第二套余额。
- 让码点数、有效期、状态、批次预算和运营操作完全由服务端控制。
- 防止明文码进入持久化数据库、普通日志、浏览器存储和分析系统。
- 为批次生成、一次性导出、暂停、撤销、冲正和审计建立最小运营边界。

**Non-Goals:**

- 不使用兑换码发放会员、知识库额度、现金、折扣券或实物权益。
- 不提供所有人共享的低熵通用口令或永久有效码。
- 不在本变更建设完整营销自动化、渠道归因、短信或微信发码系统。
- 不允许普通客服直接生成兑换码或修改积分余额。
- 不改变现有积分包价格、会员价格、AI 消费费率和支付退款规则。
- 不承诺兑换码具有现金价值、可转让性或退款能力。

## Decisions

### 1. 首发采用高熵一次性码

码使用密码学安全随机源生成，编码采用不区分大小写且排除易混淆字符的 Crockford Base32 风格，展示时分为 4 组、每组 4 位。服务端规范化时只移除允许的空格和连字符并转换大写；任何其他字符、过短或过长输入在查询摘要前拒绝。

有效随机载荷至少 80 bit。码版本和短批次提示不计入随机熵。备选的可多人共享活动码更容易泄露、穷举和超预算，且需要复杂的每账号限制，因此首发不采用。

### 2. 数据库只保存带密钥摘要

服务端使用版本化 pepper 对规范化明文执行 HMAC，例如：

```text
digest = HMAC-SHA-256(pepper[v], normalizedCode)
```

持久化 `digest` 唯一索引、`pepperVersion`、随机生成的非敏感 `publicHint` 和码状态，不保存可离线验证的普通哈希或完整明文。HMAC 密钥由服务端密钥管理提供，不能出现在数据库、客户端或日志。

备选的单向 SHA-256 虽然实现简单，但数据库泄露后可对格式空间进行离线枚举，因此不采用。加密保存完整码也会扩大解密密钥泄露影响，不作为常规存储。

### 3. 明文码只允许一次性受控导出

批次创建事务先生成码与摘要，持久化成功后把明文放入短生命周期、单次读取的加密导出对象；运营人员需要独立权限和近期重新验证才能下载。导出成功或到期后销毁对象，服务端无法再次“查看原码”，只能撤销并重新生成。

原型使用内存适配器模拟此生命周期，fixture 仅使用明显标记为 synthetic 的测试码，并不得与生产格式或密钥共用。

### 4. 使用批次和码两个生命周期层级

核心模型：

```ts
interface RedemptionCampaign {
  id: string;
  status: "draft" | "active" | "paused" | "revoked";
  pointsPerCode: number;
  codeLimit: number;
  pointsBudget: number;
  startsAtMs: number;
  expiresAtMs: number;
  redeemedCount: number;
  redeemedPoints: number;
}

interface RedemptionCodeRecord {
  id: string;
  campaignId: string;
  digest: string;
  pepperVersion: number;
  publicHint: string;
  status: "active" | "redeemed" | "revoked";
  redeemedByUserId?: string;
  redeemedAtMs?: number;
}
```

点数只来自批次，必须是受政策上限约束的正整数。码和批次都必须处于有效时间窗；批次暂停阻止新兑换但可恢复，撤销不可恢复。既有兑换历史不随暂停或撤销删除。

### 5. 兑换和钱包入账使用一个事务

兑换事务按以下顺序执行：

1. 验证登录、CSRF、格式、速率限制和幂等记录。
2. 计算 HMAC 摘要并锁定对应码与批次。
3. 校验码状态、活动时间窗、批次状态、数量和点数预算。
4. 条件更新码为 `redeemed`，写入 `Redemption` 记录。
5. 向现有钱包追加唯一 `redemption_credit` 流水。
6. 增加批次兑换计数和点数，提交事务。

`redemptionId` 同时作为账本 `referenceId`，数据库对 `codeId` 和账本 `(kind, referenceId)` 设置唯一约束。若任一步失败，事务回滚。备选的先标记码、再异步加积分会产生“码已用但未到账”的分裂状态，因此不采用。

### 6. 幂等与一次性语义分开处理

兑换请求包含账号范围内的 `idempotencyKey`，服务端保存请求结果。相同账号和键重放返回原结果。用户使用新键重复提交自己已兑换的码时，通过码记录返回原兑换摘要；另一个账号提交时只返回通用 `code-unavailable`，不泄露原用户或活动细节。

并发请求依赖数据库行锁或原子条件更新裁决；先提交的一个成功，其余失败。不能仅依靠应用进程内锁，因为多实例部署无法保证互斥。

### 7. 对用户收敛错误，对运营保留原因

用户结果分为：

- `redeemed`：到账成功。
- `already-redeemed-by-you`：当前账号此前已成功兑换。
- `code-unavailable`：格式外的无效、未开始、过期、撤销、他人已用或预算耗尽统一提示。
- `rate-limited`：显示安全的可重试时间。
- `temporarily-unavailable`：网络或服务失败，可使用同一请求重试。

服务端审计可保存更细的枚举原因，但 Web、普通客服日志和分析事件不得暴露，以减少枚举反馈。格式错误可在客户端本地提示，因为尚未查询码是否存在。

### 8. 多维限流先于兑换码查询

限流至少按用户 ID 和风险来源哈希执行，连续失败使用递增退避；成功不会重置所有风险信号。达到阈值时在摘要查询和事务之前拒绝。跨账号相似前缀、高失败率和异常地域变化只形成脱敏风险事件，后续可接入验证码或人工审核适配器。

备选的只按 IP 限流会误伤共享网络，也容易被代理绕过，因此使用账号为主、风险来源为辅的组合。

### 9. 积分页面使用内联卡片而非支付弹窗

兑换卡片放在余额与消费说明附近，包含单行输入、格式提示和“立即兑换”。提交时按钮进入忙碌状态；成功后清空输入，显示 `+N 点` 和新余额，并把新 `redemption_credit` 流水插入明细。错误保留输入以便校对，但离开页面或成功后不得写入 localStorage/sessionStorage/URL。

兑换不是支付，不进入微信/支付宝订单弹窗，不显示交易号或付款截图。移动端按钮与输入保持 44×44 CSS 像素触摸目标，状态通过文字和 `aria-live` 播报。

### 10. 运营操作使用独立权限和不可变审计

权限至少区分 `redemption-operator` 与普通客服。创建批次、生成、导出、暂停、恢复、撤销和冲正记录操作者、时间、原因、批次、数量与前后状态。审计不保存完整明文码。

欺诈冲正不修改原 `redemption_credit`；通过关联的负向 `redemption_reversal` 流水减少钱包。若用户可用余额不足，操作进入异常审核而不是生成负余额或篡改历史。

### 11. 指标不携带明文凭证

允许记录兑换成功率、通用失败类别、延迟、限流次数、批次预算利用率和账本不一致告警。用户、IP 和设备仅使用用途限定的不可逆风险哈希；`digest`、完整 code、导出内容和 HMAC 密钥均不得进入指标、普通日志或错误堆栈。

## Risks / Trade-offs

- [兑换码截图或转发后被抢兑] → 使用一次性高熵码、短有效期和先到先得语义；运营交付渠道另行评估。
- [数据库泄露后离线枚举] → 采用版本化 HMAC pepper，而不是普通哈希。
- [并发导致重复积分] → 事务锁、码唯一兑换约束和账本唯一 reference 三层保护。
- [统一错误降低客服排查效率] → 用户端使用通用错误，受权限后台通过安全短标识查看具体审计原因。
- [过严限流误伤用户] → 账号与风险来源组合、明确重试时间，并提供客服恢复路径。
- [批次配置错误导致超额发放] → 创建时校验点数上限和总预算，启用需要独立确认，事务中再次检查预算。
- [一次性导出丢失后无法恢复] → 不解密数据库；撤销未分发批次并生成新批次，保留审计。
- [冲正时积分已消费] → 不允许自动负余额，转入授权异常审核。

## Migration Plan

1. 先对齐活跃 billing、增长、产品体验 specs 与使用说明，增加兑换积分来源和安全边界。
2. 扩展协议账本类型、兑换请求/结果、批次、码状态和管理命令。
3. 实现内存型原型存储、HMAC/随机源/限流适配器与原子事务模拟，并建立并发测试。
4. 在积分页面实现兑换卡片、状态、余额和流水更新；fixture 只使用 synthetic 码。
5. 增加运营批次与一次性导出服务边界，不向普通用户 UI 暴露后台能力。
6. 运行安全、并发、账本、响应式和可访问性验证，再决定生产数据库与密钥管理实现。
7. 回滚时关闭兑换入口和批次新兑换；已入账积分与审计历史保持不变，不删除或重写账本。

## Open Questions

- 无阻塞问题。首发采用一次性积分码；共享活动码、会员兑换、定向绑定码和自动发码渠道必须通过后续独立 OpenSpec 变更。
