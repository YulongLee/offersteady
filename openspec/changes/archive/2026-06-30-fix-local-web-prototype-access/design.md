## Context

当前 Web 原型已经具备较完整的页面和交互，但“本地访问不了”这类问题会让团队误以为原型本身已经损坏。结合前一轮恢复与审查结论，这类问题往往同时涉及三种情况：本地服务未正确启动、访问路径或端口使用错误、自动巡检依赖未满足，以及少量确实会影响访问的原型回归问题。

这次变更的重点不是新增页面，而是让本地网页原型具备一条稳定、可诊断、可重复的访问链路，保证任何后续评审都能先进入正确的本地运行状态。

## Goals / Non-Goals

**Goals:**

- 修复本地网页原型打不开、打不开准、巡检误判等访问问题
- 给出清晰的本地访问顺序、预览要求和诊断边界
- 在不改变已批准原型体验的前提下，修复阻断访问的前端问题
- 让“服务未启动”“访问方式错误”“页面真实损坏”三类情况可区分

**Non-Goals:**

- 不新增业务功能，不重做页面结构
- 不改变已批准的产品文案策略和用户路径，除非它们直接导致本地访问失败
- 不在本次变更中推进正式后端联调或生产部署
- 不把“本地可访问”扩展成跨设备正式发布或外网可用方案

## Decisions

### 1. 把“本地访问链路”作为独立恢复对象处理

不把这次问题简单归类为“某个页面坏了”，而是把本地原型访问能力本身定义为需要恢复的对象。这样可以同时覆盖 dev server、preview、自动巡检和页面入口。

备选方案：

- 只修某个报错页面：快，但很可能遗漏端口、启动命令和验证顺序问题

### 2. 先验证运行链路，再判断页面是否损坏

实现时先确认 `typecheck -> test -> build -> 启动访问服务 -> 执行巡检` 这条链是否成立，再判断具体页面问题。因为当前最容易出现的误判就是在服务未启动时直接运行巡检。

备选方案：

- 直接运行自动脚本判定页面问题：会把“服务不可达”误当成“页面异常”

### 3. 优先修复访问阻断，不顺手改体验

若发现页面文案、入口或本地环境提示与访问恢复直接相关，可以修；但如果只是一般体验优化，不在这条变更里处理。

备选方案：

- 借机继续调整原型：会重新扩大范围，并增加再次偏离基线的风险

### 4. 产出诊断文档和可执行步骤

除了代码修复，还要把本地访问步骤和排查项写清楚，减少后续重复踩坑。这样同一个问题不会每次都重新靠人工口头解释。

## Risks / Trade-offs

- [风险] 用户描述“访问不了”可能指启动失败，也可能只是打开了错误地址 → [缓解] 在实施阶段先分离“服务未启动 / 地址错误 / 页面异常”三类结论
- [风险] 修复访问问题时再次修改原型行为 → [缓解] 只处理访问阻断和必要说明，体验变化必须受基线文档约束
- [风险] 自动巡检依赖本地浏览器与服务状态，环境差异可能导致偶发失败 → [缓解] 明确巡检前提，并保留构建、手动访问与脚本巡检三层验证

## Migration Plan

1. 复现当前“本地访问不了”的具体表现
2. 修复本地访问链路中的启动、端口、预览或页面阻断问题
3. 更新本地访问说明和诊断步骤
4. 重新运行本地访问验证并确认关键页面可打开

## Open Questions

- 当前“访问不了”是否表现为白屏、连接拒绝、404、脚本错误或自动跳转异常，需要在 apply 阶段先精确定位
- 是否需要增加一个更显式的本地健康检查命令，方便以后快速判断服务与页面状态
