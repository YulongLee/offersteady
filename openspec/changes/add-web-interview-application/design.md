## Context

当前 `apps/web` 是单页 React/Vite 原型，只呈现桌面伴随程序下载、配对和设备状态；`apps/desktop` 已有独立的 Mac 收音窗口。网页端尚未形成用户可理解的主流程，也没有明确区分公开站点、登录后应用、面试准备和现场工作台。

本设计建立浏览器产品的信息架构与页面状态。它依赖 `define-interview-assistant-mvp` 的资料与回答能力、`add-cross-device-interview-workspace` 的同步模型和 `add-desktop-audio-companion` 的设备协议，但不复制这些能力的底层规范。

## Goals / Non-Goals

**Goals:**

- 让首次用户能从公开首页进入一条可完成的 MVP 路径：创建面试、准备资料、检查设备、开始面试、结束复盘。
- 让进行中的面试页面始终以“当前问题与回答建议”为视觉主角。
- 清楚区分网页主产品与 Mac 收音伴随程序的职责。
- 为桌面、平板和手机浏览器定义一致但不同密度的体验。
- 覆盖空白、处理中、失败、离线、权限不足和恢复状态。
- 让敏感资料的用途、保存状态和删除操作可见。

**Non-Goals:**

- 不在本变更中实现付费、订阅、团队协作、职位推荐或企业后台。
- 不让网页绕过系统权限采集系统音频，也不把完整业务界面放进 Mac 伴随程序。
- 不在第一版提供复杂的数据分析、能力雷达或自动生成求职结论。
- 不绑定具体 AI、语音、OCR、解析器或向量数据库供应商。
- 不把隐藏、规避面试规则或不可见收音作为产品能力。

## Decisions

### 1. 一个公开入口，一个登录后应用

路由按职责分为：

| 路由 | 页面 | 核心职责 |
| --- | --- | --- |
| `/` | 产品首页 | 解释价值、使用方式、隐私边界并进入应用 |
| `/login` | 身份入口 | 登录、注册或原型访客模式 |
| `/app` | 面试首页 | 继续进行中会话、创建面试、查看最近记录 |
| `/app/interviews/new` | 新建面试 | 创建名称、岗位及使用场景 |
| `/app/interviews/:id/prepare` | 准备中心 | 简历、JD、知识库、设备及隐私检查 |
| `/app/interviews/:id/live` | 实时工作台 | 当前问题、实时回答、历史、截图和控制 |
| `/app/interviews/:id/review` | 面试复盘 | 轻量问题回顾、资料使用和删除入口 |
| `/app/library` | 资料库 | 管理可复用简历、JD 和知识材料 |
| `/app/devices` | 设备 | 下载、连接、诊断和撤销 Mac 伴随程序 |
| `/app/settings` | 设置 | 实际隐私行为、账号与辅助功能 |

选择显式路由而不是把所有功能塞入一个超长 Dashboard，因为准备态和面试态的注意力需求完全不同，独立路由也更利于恢复、深链和测试。MVP 可以使用合成数据，但路由和状态模型必须按真实产品建立。

### 2. 应用首页围绕“下一步动作”组织

登录后的首页不做数据大盘。首屏只包含：继续进行中的面试、创建新面试、最近三场记录和资料就绪摘要。没有历史时展示引导式空状态，而不是空表格。

备选方案是直接进入资料上传页；这会让用户不清楚资料属于哪场面试，也削弱多次使用时的会话模型，因此不采用。

### 3. 准备中心使用可回访的检查清单

准备中心使用资料页同一来源集合展示本场资料选择和问题输入状态。简历、JD 与知识材料均可为空；设备不可用时允许降级为手动输入和截图模式。开始条件只要求确认资料清单和一种可用输入方式，数据用途以静态说明披露，具体采集在对应操作时授权。

```text
┌──────────────────────────────────────────────────────────────┐
│ ← 面试首页   前端工程师 · 字节跳动          已完成 3/5       │
├──────────────────────────────┬───────────────────────────────┤
│ 准备资料                      │ 开始前检查                    │
│ ✓ 简历  李宇龙-前端.pdf       │ ✓ 简历已解析                  │
│ ✓ JD    高级前端工程师        │ ✓ JD 已解析                   │
│ ○ 知识库  可选                │ ○ Mac 收音程序未连接          │
│                              │ · 原始音频默认不保存          │
│ [编辑资料]                    │ [连接设备] [以手动模式开始]   │
└──────────────────────────────┴───────────────────────────────┘
```

### 4. 实时工作台采用可调整的对话与回答双栏

宽屏不再保留资料侧栏，主工作区将实时对话放在左栏，将回答、历史翻页和紧凑输入放在右栏；两栏之间提供可访问的拖动分隔条。设备状态压缩到顶部。低置信度问题在对应对话条目中确认，回答区按照“问题—回答提纲—展开详情—来源与不确定性”逐层呈现，AI 内容始终标为“回答建议”。

```text
┌─────────────────────────────────────────────────────────────────────┐
│ 面试稳  前端工程师面试        ● Mac 已连接          18:24  设置    │
├──────────────┬─────────────────────────────────────────────────────┤
│ 资料 [收起]   │ 实时对话：面试官 / 我                               │
│ ✓ 简历        ├─────────────────────────────────────────────────────┤
│ ✓ JD         │ 回答  1 / 4                         [上一条][下一条] │
│ 8 篇知识材料   │ 1. 先说明项目背景与目标                            │
│              │ 2. 聚焦个人职责与关键决策                          │
│              │ 3. 用可验证结果收尾                                │
│              │ [输入面试官的问题] [回答问题] [截图回答]            │
├──────────────┴─────────────────────────────────────────────────────┤
│ ● 面试进行中                 [暂停] [静音] [结束面试]              │
└─────────────────────────────────────────────────────────────────────┘
```

平板和手机移除分隔条并按实时对话、回答和紧凑输入单列排列；历史不拥有独立抽屉。布局变化不得重置会话、当前历史答案、输入草稿或流式回答。

### 5. 截图和手动问题是工作台内的输入方式

截图回答从主区底部的紧凑输入栏进入并通过响应式 Sheet 完成：选择/拍摄图片、预览、确认提交、识别题型、生成建议。手动问题使用相同回答管线；低置信度问题在相关对话条目内确认。三者都不得导航离开当前会话；重复提交使用客户端命令 ID 去重。

备选方案是独立截图页面；它会打断现场上下文并增加返回成本，因此不采用。

### 6. 复盘保持轻量并优先给用户数据控制

结束后展示持续时间、问题列表、回答建议和使用过的资料来源，允许用户删除整场会话或单独删除截图。首版不对候选人作人格或能力评分。若复盘摘要由 AI 生成，必须与原始问题记录分区并标明生成状态。

### 7. 使用前端路由、查询缓存和显式状态模型

实现阶段在现有 React 应用中增加客户端路由；远端资源状态与实时会话状态分离：

- 查询层处理面试列表、资料、历史和设置，并支持请求取消与错误重试。
- 会话状态机处理 `preparing`、`ready`、`active`、`paused`、`ended`、`error`。
- 实时事件层消费服务端确认的带版本事件，不将本地乐观状态冒充为已同步状态。
- AI、语音、OCR 与检索只通过服务端适配器调用，浏览器不保存供应商密钥。

原型阶段可以使用内存 fixtures，但组件只依赖领域接口，以便逐页替换为真实 API。

### 8. 统一页面状态与隐私反馈

所有数据区必须支持 `idle`、`loading`、`ready`、`empty`、`error`。实时区域额外支持 `listening`、`transcribing`、`generating`、`streaming`、`uncertain`、`offline`。状态同时使用文字和视觉标记，不能只依赖颜色。

页面标题、系统通知和未授权设备不得暴露完整简历、问题或回答。删除操作使用明确确认，并在服务端确认前显示处理中，而不是提前宣称已删除。

## Risks / Trade-offs

- [首版页面过多拖慢验证] → 实现时优先交付 `/app`、`prepare`、`live`、`review` 四个闭环页面，资料库和设置先提供最小入口。
- [当前下载页与新应用导航割裂] → 将下载/连接能力迁移为 `/app/devices`，首页仅保留轻量下载入口。
- [实时状态与普通请求状态混杂] → 使用独立的查询缓存、会话状态机和实时事件层。
- [三栏布局在小屏不可用] → 平板使用抽屉，手机使用单栏与底部导航，并保持同一会话状态。
- [AI 建议看起来像候选人真实经历] → 回答卡区分原始资料引用、AI 推断和生成建议；缺少依据时禁止生成具体履历事实。
- [原型 fixtures 形成不可替换的页面逻辑] → 定义领域接口和 fixture adapter，不让组件直接导入静态业务数据。
- [敏感内容被浏览器缓存或日志记录] → 响应使用适当缓存策略，前端日志仅记录状态与错误码，删除流程覆盖服务端和客户端缓存。

## Migration Plan

1. 保留当前页面作为 `/app/devices` 的视觉基线，建立路由和应用外壳。
2. 使用合成数据实现 `/app`、`prepare`、`live`、`review` 四页可点击原型。
3. 完成桌面、平板和手机响应式验收，以及键盘和触控可用性测试。
4. 依次接入面试草稿、资料状态、设备状态、实时事件和复盘 API。
5. 最后接入真实 AI、语音、OCR 和检索适配器，并执行脱敏评测与隐私检查。

若实时接口未准备好，保留 fixture adapter 和手动输入模式，不回退路由与页面架构。

## Open Questions

- 原型首版是否需要真实账号登录，还是先提供仅本机的访客模式？
- 面试记录自动到期删除策略需要等服务端机制和合规评审明确后另行设计；设置页当前不提供期限偏好。
- 手机网页首版只承担查看、暂停和拍题，还是也允许作为麦克风输入设备？
- 公共首页首版需要完整营销内容，还是只保留产品说明和进入应用按钮？
