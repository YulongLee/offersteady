## Why

截屏回答是面试稳处理线上笔试、代码题、系统设计题的关键入口，但当前用户只能感知“等待答案”，无法区分截图上传、OSS、视觉模型和答案保存分别耗时多少。现在已经确认最终答案主要受视觉模型 API 波动影响，因此下一轮优化应聚焦在不打断全屏使用习惯的阶段反馈、分段 telemetry 和可复测性能报告。

## What Changes

- 为远程截屏回答增加分段耗时采集，覆盖桌面压缩、上传返回、OSS 写入、签名 URL、视觉模型、答案保存、前端轮询完成等阶段。
- 在面试页截屏回答 UI 中展示明确阶段：等待桌面、截图上传中、已上传/识别中、生成答案中、完成、失败。
- 保持整屏截图，不引入手动框选或裁剪交互。
- 保持 `qwen3.6-flash` 作为视觉回答模型，不把更换模型作为本次优化目标。
- 保持截屏回答不使用简历、JD 或知识库 RAG，避免线上笔试/代码题答案被个人资料污染。
- 增加本地自测脚本或命令，使用指定图片跑完整链路并输出每个阶段耗时报告。
- 不保存原始截图到诊断报告；诊断只记录尺寸、字节数、阶段、耗时、模型名、错误码、request/task id。

## Capabilities

### New Capabilities

- None.

### Modified Capabilities

- `remote-screenshot-answer-pipeline`: 增加截屏回答阶段进度、分段性能 telemetry、非阻塞上传体验和可复测性能报告要求。

## Impact

- Backend: `apps/backend` 的 screenshot answer service、remote capture request response、后台任务状态、性能指标记录和测试脚本。
- Web: `apps/web` 面试页截屏回答弹窗/答案区域的阶段展示和失败提示。
- Desktop: `apps/desktop` 保持整屏压缩上传链路，并上报压缩尺寸/字节数等非敏感元数据。
- AI: 保持现有视觉模型和提示词方向；只记录模型耗时，不更换模型。
- Privacy: 诊断报告不得包含原始截图、base64 图片、答案之外的敏感图像内容；截图对象仍按临时 OSS 对象和既有生命周期策略处理。
