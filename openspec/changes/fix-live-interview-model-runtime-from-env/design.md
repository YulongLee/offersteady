## Context

当前产品已经具备“创建面试 → 确认资料 → 进入面试页 → 手动输入问题 / 快答”的页面路径，也已经存在后端 `live-answer`、`ChatService` 和 Qwen-compatible gateway。但从真实使用反馈看，进入面试页后依然会出现“模型当前不可用”或“回答生成失败”，即使 `.env` 中已经填好了模型配置。

这说明问题不只是前端按钮没连上，而是整条运行时链路中至少有一个环节没有稳定对齐：

- 后端是否真的读取了当前 `.env`
- 读取到的配置是否能进入 `ChatService`
- 运行时是否错误落回了测试 / synthetic 分支
- 返回给前端的错误是否足够区分“会话未启动”和“模型配置失败”

本变更聚焦“实时面试页手动问题回答”这一条链路，不改页面布局，不扩展截图回答、实时语音、支付或登录。

## Goals / Non-Goals

**Goals:**

- 让实时面试页手动问题 / 快答稳定调用后端真实模型链路
- 保证后端从服务端 `.env` 读取模型配置，而不是依赖前端或测试分支
- 对模型不可用问题进行明确分类，并把安全错误原因返回到页面
- 保留原问题和重试能力，避免前端伪造成功回答
- 为本地联调提供可重复验证的诊断路径

**Non-Goals:**

- 不修改实时面试页主布局与操作位置
- 不新增新的模型供应商，只拉通当前已接入的 Chat runtime
- 不改截图回答、实时语音、支付、积分、微信登录
- 不在本次变更中重做完整流式协议

## Decisions

### Decision 1: 以后端运行时为唯一模型配置事实源

实时问答所需的 `base_url`、`api_key`、`model` 全部由后端配置层统一读取，前端只负责向 `live-answer` 发送问题和会话标识。

Rationale:

- 密钥不能进入客户端
- 页面不应该自己判断模型是否配置
- 同一套配置也能被集成验证和后端测试复用

Alternatives considered:

- 前端读取公开变量判断模型可用性：会造成错误判断，而且密钥边界不安全
- 在每个 service 内各自直接读原始环境变量：会让排查路径分散

### Decision 2: 保留现有 live-answer API，补强错误分类而不是新开接口

本次不新增新的“模型探活专用问答接口”，而是在现有 `POST /api/v1/live-answer/questions` 和相关状态路径内补齐错误分类与安全摘要。

Rationale:

- 用户真实操作就是点快答，最有价值的是把这条链路打通
- 现有前后端契约已经存在，改动更小
- 避免额外接口和真实调用行为脱节

Alternatives considered:

- 新增 `/api/v1/live-answer/readiness` 专用接口：可做，但不能替代真实问答链路，只能作为辅助

### Decision 3: 区分“会话问题”和“模型问题”

后端需要把以下错误边界明确拆开：

- Session 未启动 / 非 live
- 模型配置缺失
- 模型鉴权失败
- 供应商不可达 / 超时 / 限流
- 供应商响应体无效

前端收到后直接展示安全摘要，并保留问题文本用于重试。

Rationale:

- 这次用户反馈的核心就是“明明配了 `.env` 还是不可用”，必须让错误可定位
- 如果继续把不同失败都压成“回答生成失败”，联调效率会很低

Alternatives considered:

- 只在日志里区分错误，前端继续统一提示：对开发和用户都不够友好

### Decision 4: 明确禁止实时面试页回退到本地假成功回答

当后端模型运行时失败时，前端可以展示 pending、failed、retry，但不能本地拼一条“看起来像成功”的答案填进右侧回答区。

Rationale:

- 这会掩盖真实链路问题
- 会让用户误判模型已经可用
- 不利于后续 E2E 联调

## Risks / Trade-offs

- [Risk] 不同环境下 `.env` 加载路径不一致，导致“本地能测、页面不能用” → Mitigation: 把本次变更绑定到现有 config 层和本地联调文档，而不是在业务代码里临时兜底
- [Risk] 错误分类变多后，前端提示文案可能不一致 → Mitigation: 由后端返回安全错误摘要，前端只做有限映射
- [Risk] 真实模型调用成功，但页面仍显示失败，问题可能在前端状态同步 → Mitigation: 同时验证 live-answer API 契约、adapter 映射和页面回归测试
- [Risk] 增加运行时诊断后误暴露敏感信息 → Mitigation: 只输出错误类别、request id、模块名和安全摘要，不输出密钥、完整 Prompt 或原始用户材料

## Migration Plan

1. 审计后端 `Settings -> ChatService -> QwenCompatibleGateway -> live-answer module` 的配置注入路径。
2. 补充 live-answer 运行时错误分类和安全消息映射。
3. 审计前端快答调用链，确认不再存在本地 synthetic success fallback。
4. 增加 `.env` 配置成功、缺失、鉴权失败、供应商失败的回归验证。
5. 更新必要文档与集成验证说明。

Rollback strategy:

- 如果本次分类或诊断改动引入额外问题，可回滚到当前 live-answer 契约；
- 但不应恢复为“前端假成功回答”模式，最多只保留真实失败提示。

## Open Questions

- 是否需要补一个轻量 `status/readiness` 端点，方便页面在进入面试前预检模型可用性？本变更不依赖它，但可以作为后续优化。
- 供应商级别的错误文案是否要进一步归一成统一 error code 枚举，以便前端和 integration report 共用？
