## Context

当前准备页已经允许资料为空或部分选择，但“开始面试”仍依赖“至少一种问题输入方式可用”。这让用户在完成“创建面试 → 确认资料”后，还需要处理一个 Web 端输入检查，容易误以为必须先连接或授权本地端才能进入面试页。

产品职责应拆开：Web 准备页负责确认本场资料；本地端软件负责检查麦克风、系统音频、声道、问题检测等输入能力；实时页负责在具体操作发生时呈现手动输入、截图回答、音频状态和错误恢复。

## Goals / Non-Goals

**Goals:**

- 让用户确认本场资料后即可点击“开始面试”进入实时页。
- 移除 Web 准备页中对问题输入方式的阻塞检查。
- 把音频、收音、问题检测可用性检查明确交给本地端软件和实时页状态。
- 保持现有页面结构、实时页布局和主要操作不变。

**Non-Goals:**

- 不新增真实语音识别、截图识别或 Chat 业务逻辑。
- 不改变资料选择规则；未就绪、删除、失败资料仍不可选。
- 不默认开启麦克风、系统音频或截图上传。
- 不修改收费策略、登录方式或会员体系。

## Decisions

### 1. Web 准备页只 gate “资料清单确认”

开始按钮启用条件从 `selectionReady && (audioReady || manualMode)` 调整为 `selectionReady`。如果资料选择未确认，继续禁用并提示确认本场资料；如果资料已确认，即使本地端未连接或音频未授权，也允许进入实时页。

替代方案：保留手动模式按钮作为必选步骤。该方案仍会把“问题输入准备”变成 Web gate，不符合用户希望“确定好资料后就能进入面试”的流程。

### 2. 输入检查降级为非阻塞状态说明

准备页可以继续展示“本地端负责检查收音和问题识别”“未连接时仍可进入后使用手动输入/截图回答”等说明，但这些说明不能禁用开始按钮。实时页继续根据设备状态显示“等待开始面试”“需要音频权限”“设备连接异常”等具体状态。

替代方案：完全隐藏输入检查。这个方案会让用户不知道为什么自动收音没有工作；保留非阻塞说明更利于调试。

### 3. 本地端软件成为输入就绪诊断来源

本地端下载软件在连接后报告麦克风、系统音频、声道来源和问题检测可用性。Web 只展示这些状态，不在准备页自行判断是否可以进入面试。自动回答只有在本地端报告可用并收到问题事件后触发；否则实时页保留手动输入和截图回答。

替代方案：Web 自己检查浏览器麦克风和系统音频。该产品需要桌面伴随程序处理系统音频和双声道识别，浏览器检查无法覆盖关键能力。

## Risks / Trade-offs

- [用户进入实时页后发现本地端不可用] → 在准备页和实时页显示明确的非阻塞提示，并保留手动输入、截图回答。
- [误解为点击开始面试就开始收音] → 文案必须说明进入面试页不等于授权或启动音频采集；收音仍由本地端授权和实时页“开始面试/开始采集”状态控制。
- [后端 session start 仍有输入 gate] → 实现时同步检查后端启动接口，确保只校验会话权限、资料确认和资料有效性。

## Migration Plan

1. 调整准备页完成度和开始前检查文案。
2. 修改开始按钮启用条件为资料清单已确认。
3. 更新测试，覆盖“资料确认后即使无本地端/音频权限也可进入实时页”。
4. 更新用户旅程和使用说明中“准备问题输入方式后开始”的旧描述。
5. 若需要回滚，只恢复前端 gate 逻辑；不得把通用数据用途复选框或问题输入检查重新作为资料确认的一部分。
