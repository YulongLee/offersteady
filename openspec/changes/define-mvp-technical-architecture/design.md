## Context

OfferSteady 已经形成了较清晰的 MVP 产品轮廓：用户通过网页进入产品，管理简历、JD 和知识资料，开始面试后在实时工作台中查看面试官与“我”的对话，以及 AI 给出的回答建议；在需要时，用户还可以上传截图处理笔试或系统设计问题。为了提升真实面试场景的适配度，产品同时规划了桌面伴随程序，用于在受用户明确授权时采集本地麦克风和系统音频，并把结果桥接到同一场 Web 会话。

当前仓库已经有若干原型代码与能力规范，但它们更多是围绕单个功能切片展开：Web 页面、跨端会话、桌面收音、积分钱包、兑换码等都在推进中，尚缺一份把这些模块串起来的整体技术架构设计。与此同时，项目仍处于 MVP 验证阶段，技术栈、部署环境和第三方供应商都没有最终定案，所以这份设计不能直接写死具体框架或云服务，而应该优先定义稳定的系统分层、接口边界和演进约束。

这份 Architecture Design 面向三个主要使用者：

- 产品与设计：确认哪些体验由网页承担，哪些能力需要桌面伴随程序，哪些属于后续阶段
- 工程：明确代码应该落在哪一层，哪些依赖必须通过可替换适配器接入
- 后续安全与运营：理解资料、音频、截图、积分和兑换码等敏感数据如何流经系统

## Goals / Non-Goals

**Goals:**

- 定义 OfferSteady MVP 的统一逻辑架构，包括 Web、桌面伴随程序、API、AI 编排、资料处理、计费和存储边界
- 说明核心用户链路如何穿过这些组件，包括资料导入、开始面试、实时回答、截图问答、跨端恢复和积分消费
- 约束敏感数据、日志、凭证和删除能力的默认处理原则
- 为未来代码实现提供目录与模块划分建议，但不提前锁定具体前端框架、后端语言、数据库或云厂商
- 标记哪些能力属于 MVP 必需，哪些属于生产化加固或后续产品演进

**Non-Goals:**

- 不实现任何业务代码、基础设施代码或部署脚本
- 不最终决定具体模型供应商、语音服务、向量数据库、支付渠道 SDK 或消息队列产品
- 不承诺多区域、多活、高可用生产架构
- 不把当前架构说明扩展为完整的安全合规或法律意见文档

## Decisions

### 1. Use a four-layer MVP architecture with shared domain contracts

MVP 采用四层逻辑架构：

1. Experience Layer
   Web 应用承担公开首页、登录绑定、资料管理、计费与实时面试工作台；桌面伴随程序承担设备能力接入，例如麦克风、系统音频、权限引导和与 Web 会话的桥接。
2. Application Layer
   应用 API 负责账号、会话、资料、回答任务、截图任务、计费、兑换和运营命令的统一入口，并对客户端暴露稳定协议。
3. Intelligence & Processing Layer
   AI 编排、文档解析、知识索引、转录路由、回答生成、截图分析都作为服务端受控流程存在，经由可替换适配器调用第三方能力。
4. Data & Governance Layer
   持久化状态、对象存储、审计日志、评测资产、删除任务和配置元数据统一归于这一层。

跨层共享的契约优先放在 `packages/protocol` 或未来共享领域模块中，避免 Web、桌面和 API 分别定义自己的事件形状。

选择这个方案，是因为它能把“用户体验载体”和“AI/供应商接入”拆开，也能允许 MVP 先用简单服务实现，再逐步替换底层适配器。替代方案是直接围绕单个前端应用组织所有逻辑，例如把转录、计费和回答状态都固化在 Web 层；这种方式虽然上手快，但会让桌面伴随程序、跨端会话和后续 API 难以共享约束。

### 2. Treat the Web application as the primary product surface, with desktop as a scoped companion

MVP 以 Web 应用作为统一产品入口，负责账号绑定、资料管理、购买与积分、会话列表、准备页和实时工作台。桌面端不是第二套独立业务系统，而是一个被会话授权的 companion，用于补足浏览器难以稳定完成的设备采集能力。

桌面伴随程序只负责：

- 设备权限与状态展示
- 麦克风与系统音频采集
- 将音频帧、转录片段或桥接事件发送到同一场会话
- 显示最小必要的连接与异常信息

桌面端不承担：

- 独立资料管理
- 独立计费系统
- 独立回答历史或知识库编辑

这样做可以让手机端、平板端和普通桌面浏览器都共享同一套产品体验，同时把本地系统权限收敛在伴随程序内。替代方案是做“桌面主应用 + Web 辅助页面”，但这会抬高首发门槛，也不利于非桌面场景验证。

### 3. Separate synchronous interview interactions from asynchronous processing jobs

架构上把用户面前的同步交互，与后台可排队或可重试的异步任务拆开：

- 同步链路：会话进入、实时对话流、手动提问、回答生成、终止回答、工作台状态恢复
- 准同步链路：截图上传后的单次分析与回答返回
- 异步链路：简历/JD/知识资料解析、索引构建、批量删除、运营导出、风控审计聚合

实时回答不应依赖“所有资料后台处理完成”这一前提；如果某些资料尚未就绪，系统应在会话上下文中显式降级，只使用已确认可用的来源。这样既能缩短 MVP 链路，也更符合你前面已经定义的“本场资料可为空、可按状态选择”的产品行为。

替代方案是让所有资料都必须先完成索引才允许开始会话。这个方案实现更简单，但会让首次使用等待过长，也和当前产品流程不一致。

### 4. Keep state ownership explicit: ephemeral UI state on clients, authoritative session state on the server

客户端只持有短生命周期的界面状态，例如输入草稿、当前滚动位置、本地折叠状态、临时上传预览和一次交互中的 pending 标记。会影响跨设备、跨刷新一致性的状态统一以服务端为准，包括：

- 账号、会话、资料与版本
- 本场资料选择
- 回答历史与回答任务状态
- 截图任务状态
- 钱包余额、会员权益与兑换流水
- 桌面桥接连接关系与活跃会话标识

这样定义的原因是，OfferSteady 已经明确支持手机端与电脑端共享同一网页入口，也会出现桌面伴随程序与 Web 同步的组合。只靠本地状态无法支撑“刷新后恢复”与“多端一致”。替代方案是尽量把所有状态保存在浏览器或桌面本地再做同步，这会增加冲突处理复杂度，而且不利于权限与删除控制。

### 5. Standardize AI orchestration behind task-oriented interfaces

AI 相关能力不直接散落在 UI 或控制器里，而是通过明确的任务接口组织：

- `prepare-materials`
- `index-knowledge-source`
- `transcribe-live-audio`
- `generate-live-answer`
- `analyze-screenshot-question`
- `summarize-interview-review`

每个任务接口接收统一的输入上下文，返回结构化结果、来源标记、不确定性和版本信息；对应 Prompt 模板与评测案例分别放在 `ai/prompts` 与 `ai/evals`。这既符合仓库现有工程原则，也能避免后续换模型时大面积改 UI 逻辑。

替代方案是让每个页面直接拼 prompt 并调用模型。这样做在原型期很快，但无法满足后续可评测、可审计和可替换的要求。

### 6. Design sensitive data handling around minimum retention and explicit deletion

对 MVP 中的敏感数据，架构采用“最小保存 + 明确入口 + 分类处理”的原则：

- 简历、JD、知识材料：可持久化，但要记录来源、版本、处理状态和删除入口
- 截图：仅在用户明确提交后进入服务端，支持查看与删除
- 原始音频：默认不长期保存；如需短暂缓冲，应作为临时处理数据，不作为长期业务资产
- 转录片段与回答：按会话记录管理，可删除并受会话权限控制
- 积分兑换码和其他 bearer secret：不得进入 URL、普通日志、分析事件、客户端持久化存储

这不是为了把所有数据都拒之门外，而是为了让产品在真实面试辅助场景下先建立可信边界。替代方案是统一把所有原始输入长期存档，虽然利于复盘和调试，但与当前隐私边界和用户信任目标冲突。

### 7. Use modular storage choices rather than one database assumption

虽然这份设计不锁死具体基础设施，但会把存储按职责拆分：

- Transactional Store：账号、资料元数据、会话、回答任务、钱包、兑换与审计记录
- Object Store：截图、资料原文件、短时导出内容
- Search / Retrieval Store：知识索引、片段引用和检索元数据
- Evaluation & Prompt Assets：Prompt 版本、评测用例、脱敏样本

MVP 可以先用相对简单的单区部署和有限种类存储来承载这些职责，但接口上不要把“关系型表结构”“对象键名”直接暴露给客户端或上层业务。

替代方案是所有内容先塞进单一数据库或本地文件。原型阶段能工作，但后续在截图、索引和审计上会很快出现边界混乱。

### 8. Make observability and audit first-class for billing, AI, and session control

MVP 架构要从第一天就预留以下观测面：

- 会话级事件：进入准备、开始面试、停止回答、截图回答、离开会话
- AI 任务级事件：任务类型、耗时、模型版本、结果状态、不确定性标记
- 计费级事件：积分预留、积分消费、会员抵扣、兑换入账、兑换冲正
- 安全级事件：权限失败、限流、导出、删除、敏感操作审计

这里的重点不是要上复杂数据平台，而是要约束“记录什么”“不能记录什么”，尤其是不得把原始敏感文本、完整兑换码和密钥材料直接打进普通日志。

替代方案是先不做审计和观测，等产品成熟再补。这通常会让后续定位计费与回答问题变得非常困难。

### 9. Deliberately defer production hardening behind clear adapter boundaries

架构文档会明确以下内容属于后续生产化而不是当前 MVP 必需：

- 多实例实时会话协调
- 分布式限流与共享风控信号
- KMS 托管密钥与 pepper
- 任务队列与工作流编排平台
- 多区域容灾和高级备份恢复
- 自动支付回调、自动发码与更复杂运营系统

这类延后不是忽略，而是要求所有相关模块先通过端口和适配器建模，确保未来替换时不推翻上层产品行为。

## Risks / Trade-offs

- [Risk] Web 为主、桌面为辅会让部分系统音频能力依赖额外下载
  Mitigation: 在架构中把桌面桥接限定为 companion，Web 端仍可支持手动提问、截图回答和无伴随程序的基础体验

- [Risk] 不锁死技术栈可能让实现阶段再次讨论过多
  Mitigation: 先锁定模块职责、协议边界和状态归属，把具体框架选择留到对应实现变更中决策

- [Risk] 把实时链路与异步处理拆开会增加接口数量
  Mitigation: 使用统一协议层和任务接口命名，减少“每个功能一套事件模型”的混乱

- [Risk] 默认最小保存会降低调试和复盘便利性
  Mitigation: 用可删除的会话级记录、脱敏审计和明确的临时处理日志替代长期保存原始音频

- [Risk] 资料、会话、计费和 AI 编排分层后，MVP 看起来比纯前端原型更复杂
  Mitigation: 说明哪些模块可以先用简单单体服务或内存/单库适配器承载，避免把逻辑分层误解成必须微服务化

## Migration Plan

1. 先将这份架构设计作为 MVP 的统一参考，供后续各功能 change 在 proposal / design 中引用。
2. 基于该设计，增补或整理 `docs/architecture.md` 和相关 ADR，使仓库中的长期事实与当前架构说明一致。
3. 后续每个实现型变更只实现自己范围内的切片，例如 Web 工作台、桌面桥接、资料处理、计费系统，而不是一次性铺开全部模块。
4. 当某个模块要进入生产化阶段时，再通过单独 OpenSpec change 决定数据库、队列、KMS、支付回调或部署方案。

回滚策略：本变更不涉及运行时代码，只涉及架构文档与规范，因此回滚方式是撤销文档或用新的变更替换对应架构决策。

## Open Questions

- 首发时，桌面伴随程序是否只桥接音频，还是还要承担截图快捷上传与悬浮控制入口？
- 实时回答链路是以“转录文本流”为主，还是要保留更细的音频分段事件模型？
- 知识资料索引在 MVP 中需要达到什么延迟目标，是否允许先做同步小文件处理、后续再异步化？
- 计费消费在回答生成中的扣费时点，最终以“开始生成”“生成成功”还是“用户确认采用”为准？
- 微信登录、支付、客服与兑换运营后台，是否在 MVP 中都由同一个应用 API 承载，还是需要预留单独后台边界？
