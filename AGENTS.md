# OfferSteady Codex Guide

## 项目阶段

本项目目前处于产品原型阶段。优先验证用户流程和核心价值，不提前引入生产级复杂度，也不擅自确定尚未批准的技术栈。

## 开始任务前

1. 阅读根目录 `README.md`。
2. 阅读与任务对应的 `openspec/changes/<change>/`；维护既有行为时同时阅读 `openspec/specs/<capability>/`。
3. 阅读任务直接涉及的 `docs/` 文档。
4. 检查 Spec 状态、非目标、验收标准和待确认问题。
5. 如果关键信息缺失且不同选择会显著改变产品行为，先询问用户。

## Spec 驱动规则

- 新功能、显著行为变化和架构变更必须通过 `openspec/changes/` 管理。
- `proposal.md` 说明动机和范围；能力 `spec.md` 描述可测试行为；`design.md` 描述实现方案；`tasks.md` 描述执行顺序。
- `specs/` 是迁移前的手工草案，只作历史参考；不要继续在那里创建新功能。
- OpenSpec 变更未达到 apply-ready 或未获得用户确认时，默认只允许研究、澄清和原型工作，不进入生产实现。
- 每次只实现用户批准范围内的任务，不顺手扩展功能。
- 实现与 Spec 冲突时，停止扩大实现并指出冲突；不能为了适配代码而静默改变需求。
- 完成任务后，更新任务状态，运行 OpenSpec 校验并逐项核对能力场景。

## 工程原则

- 优先选择简单、可替换的设计。
- 不在客户端保存服务端密钥。
- 默认不持久化面试音频、转录或个人信息。
- 真实用户数据不得进入测试夹具；样本必须合成或脱敏。
- Prompt 统一存放在 `ai/prompts/`，不得散落在 UI 或业务代码中。
- AI 行为变更必须同时补充或更新 `ai/evals/` 中的评测案例。
- 缺陷修复应增加能够复现问题的回归测试。
- 未确定技术栈前，不创建仅用于占位的依赖和构建配置。

## 验证要求

代码工具链建立后，应在这里记录准确的安装、格式化、检查、测试和构建命令。Codex 不得虚构已经执行的验证；无法运行时需要明确说明原因。

当前文档阶段的最低验证：

- Markdown 链接指向现有文件。
- 活跃 OpenSpec 变更通过 `openspec validate <change> --strict`。
- Proposal 中列出的每项能力都有对应 Spec。
- Specs 中没有未标注的假设。

## 文档维护

- 长期产品事实放在 `docs/`。
- 正在开发的需求和决策放在 `openspec/changes/`，已确认的当前行为放在 `openspec/specs/`。
- 重复发生的 Codex 误解或评审意见，应精炼后补充到最近作用域的 `AGENTS.md`。
- 保持本文件简短，只记录反复适用的项目规则。
